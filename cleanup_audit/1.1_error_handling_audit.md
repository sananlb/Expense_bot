### 1. Bare except (9 случаев) - ВЫСОКИЙ ПРИОРИТЕТ

Использование `except:` без указания типа исключения - **плохая практика**, ловит все ошибки включая системные (KeyboardInterrupt, SystemExit).

#### 1.1 `bot/services/analytics_query.py:509`
```python

    if operation_type == 'expenses':
        category = ExpenseCategory.objects.get(id=category_id)
    else:  # incomes
        category = IncomeCategory.objects.get(id=category_id)
    result['category'] = category.get_display_name(user_lang)
except:  # ❌ BARE EXCEPT
    result['category'] = item.get('category__name', '')
```
**Проблема:** Проглатывает DoesNotExist и другие ошибки БД без логирования
**Риск:** Тихая деградация данных, сложность отладки
**Решение:** `except (ExpenseCategory.DoesNotExist, IncomeCategory.DoesNotExist, AttributeError) as e:` + логирование

## 🟡 ВАЖНЫЕ ЗАМЕЧАНИЯ

### 1. Дублирование кода обработки ошибок

**Пример:** Получение языка пользователя повторяется 3 раза в `expense_functions.py`:
```python
# Строки 131, 233, 1508 - ОДИНАКОВЫЙ КОД
try:
    profile = Profile.objects.get(telegram_id=user_id)
    lang = profile.language_code or 'ru'
except:
    pass
```

**Решение:** Создать утилиту:
```python
# bot/utils/language.py
def get_user_language_safe(user_id: int, default: str = 'ru') -> str:
    """Safely get user language with fallback"""
    try:
        profile = Profile.objects.get(telegram_id=user_id)
        return profile.language_code or default
    except Profile.DoesNotExist:
        logger.debug(f"Profile not found for user {user_id}, using default lang: {default}")
        return default
    except Exception as e:
        logger.warning(f"Error getting language for user {user_id}: {e}")
        return default
```

### 4. Логирование с `exc_info=True` используется редко

При логировании исключений полезно включать stack trace:
```python
# Текущий подход (738 случаев):
logger.error(f"Error: {e}")

# Лучший подход для критичных ошибок:
logger.error(f"Error: {e}", exc_info=True)  # Добавляет stack trace
```

**Где использовать `exc_info=True`:**
- БД ошибки (IntegrityError, DataError)
- Ошибки внешних API (OpenAI, Google)
- Критичные операции (платежи, экспорт)

---

### 5. Смешивание sync и async в обработке ошибок

Некоторые функции async, но обработка ошибок sync:
```python
async def some_function():
    try:
        sync_function()  # ❌ Блокирует event loop
    except Exception as e:
        logger.error(f"Error: {e}")
```

**Решение:** Использовать `sync_to_async` где необходимо
