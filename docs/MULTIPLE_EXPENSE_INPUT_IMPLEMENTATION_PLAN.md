# План реализации множественного ввода трат

## Цель

Добавить возможность создавать несколько трат из одного сообщения.

Основные сценарии:

```text
мороженое 120, помидоры 232, хлеб 20
```

```text
мороженое 120 помидоры 232 хлеб 20
```

Второй сценарий особенно важен для голосового ввода, потому что распознавание речи часто возвращает текст без запятых и переносов строк.

Подтверждение у пользователя не запрашиваем. Если парсер решил, что это несколько трат, создаем их сразу.

## Принципы

1. Множественный ввод должен быть отдельным слоем перед текущим одиночным парсером.
2. Текущий `parse_expense_message()` остается основным источником правды для разбора одной траты.
3. Batch-парсер решает только одно: нужно ли разбить сообщение на несколько отдельных записей.
4. Batch-парсер не валидирует суммы и категории. Он возвращает только строки отдельных трат.
5. Для голосового ввода допускается эвристика без явных разделителей.
6. Для текстового ввода пробел сам по себе не является batch-разделителем.
7. Явный разделитель всегда разделяет ввод. Одна сумма должна быть написана вместе: `48000`, не `48 000`.

## Новый модуль

Создать файл:

```text
bot/utils/multiple_expense_parser.py
```

API:

```python
def split_multiple_expense_texts(
    text: str,
    input_source: str = "text",
) -> list[str] | None:
    ...
```

Возвращает:

- `list[str]`, если сообщение разделено минимум на две части;
- `None`, если сообщение нужно обрабатывать текущим одиночным сценарием.

## Источник ввода

Расширить `handle_text_expense()` параметром:

```python
input_source: str = "text"
```

Для обычных текстовых сообщений использовать:

```python
input_source="text"
```

В `handle_voice_expense()` передавать:

```python
await handle_text_expense(
    message,
    state,
    text=voice_text,
    lang=lang,
    input_source="voice",
)
```

## Правила для текстового ввода

Для `input_source="text"` batch включается только при явных разделителях:

- запятая;
- точка с запятой;
- перенос строки.

Пример:

```text
мороженое 120, помидоры 232, хлеб 20
```

Результат:

```text
мороженое 120
помидоры 232
хлеб 20
```

Запятая между цифрами тоже считается разделителем. Одна сумма должна быть написана вместе:

```text
кофе 120,50
```

Результат:

```text
кофе 120
50
```

## Правила для голосового ввода

Для `input_source="voice"` порядок такой:

1. Сначала применить правила явных разделителей.
2. Если явных разделителей нет, попробовать voice-batch эвристику.

Voice-batch эвристика ищет повторяющийся шаблон:

```text
описание сумма описание сумма описание сумма
```

Пример:

```text
мороженое 120 помидоры 232 хлеб 20
```

Результат:

```text
мороженое 120
помидоры 232
хлеб 20
```

## Числа внутри названий

Числа внутри названий не должны сами по себе создавать новые траты.

Пример без следующей траты:

```text
Ленина 5 255 руб
```

Это не batch. Сообщение должно уйти в одиночный парсер.

Пример со следующей тратой:

```text
Ленина 5 255 руб хлеб 20
```

Результат:

```text
Ленина 5 255 руб
хлеб 20
```

Дальше каждую часть разбирает одиночный парсер. Одиночный парсер должен понимать `Ленина 5 255 руб` как `Ленина 5` на `255 руб`, а не как `5255 руб`.

Еще пример:

```text
айфон 15 80000 чехол 1000
```

Результат:

```text
айфон 15 80000
чехол 1000
```

## Интеграция в обработчик

В `bot/routers/expense.py`, перед одиночным вызовом:

```python
parsed = await parse_expense_message(...)
```

добавить:

```python
multiple_texts = split_multiple_expense_texts(
    text,
    input_source=input_source,
)
```

Если `multiple_texts` содержит две или больше частей:

1. Каждую часть разобрать текущим `parse_expense_message(..., use_ai=True)`.
2. Для каждой распарсенной позиции получить или создать категорию через текущий `get_or_create_category()`.
3. Создать трату через текущий `add_expense_with_conversion()`.
4. Сформировать отдельное сообщение через `format_expense_added_message()`.
5. Отправить отдельное сообщение на каждую трату с кнопкой редактирования.
6. Вызвать текущие уведомления лимитов через `send_expense_limit_alerts()`.
7. Завершить обработчик, не переходя в одиночный сценарий.

## AI и категории

Для первой версии каждую часть сразу разбираем через `parse_expense_message(..., use_ai=True)`, чтобы batch и одиночный ввод имели одинаковое поведение категорий, валют и описаний.

Если на длинных списках будет медленно, можно оптимизировать позже:

- использовать AI только для элементов без категории;
- ограничить параллельность;
- добавить быстрый keyword-only режим.

## Лимиты

Добавить лимит на количество создаваемых трат из одного сообщения.

Рекомендуемый лимит:

```python
MAX_MULTIPLE_EXPENSES_PER_MESSAGE = 10
```

Если парсер нашел больше 10 элементов, использовать только первые 10 или вернуть `None`.

Более безопасный вариант для первой версии: вернуть `None`, чтобы не создать лишние записи из плохого распознавания речи.

## Ошибки

Подтверждение не запрашиваем.

Если одна из трат не создалась из-за `ValueError`, можно:

1. остановить обработку и отправить ошибку;
2. или создать валидные, а по невалидным отправить краткое сообщение.

Для первой версии лучше остановить всю batch-операцию до создания записей, если на этапе validation есть ошибка. Это снижает риск частично созданного списка.

Если splitter уже вернул несколько частей, не откатываемся к одиночному парсеру исходной строки. Все части сначала должны распарситься одиночным парсером; если хотя бы одна часть не распарсилась, batch останавливается с короткой ошибкой и не создает частичный список.

## Отдельная связанная задача

Одиночный парсер должен быть исправлен в рамках этой же реализации, потому что batch теперь только разбивает текст, а итоговое поведение определяется одиночным парсером.

Кейс:

```text
Ленина 5 255 руб
```

Нельзя разбирать как:

```text
Ленина — 5255 руб
```

Ожидаемый результат:

```text
Ленина 5 — 255 руб
```

Пробел в общем случае больше не считается разделителем тысяч. Исключение: если
группа после пробела — ровно `000`, она приклеивается к числу. Это покрывает
самый частый русский формат круглых сумм:

```text
48 000      => 48000
50 000      => 50000
100 000     => 100000
1 000 000   => 1000000
```

Не-`000` группа после пробела по-прежнему НЕ склеивается (остаётся отдельным
числом / уходит в описание):

```text
1 500   => 500 (а "1" в описании)
5 255   => 255 (а "5" в описании, кейс "Ленина 5 255")
```

Реализовано через альтернативу `(?:,\d{3}|\s000)` в группах тысяч
(`bot/utils/expense_parser.py`, `AMOUNT_PATTERNS`, `_extract_leading_amount`,
fallback) и `(?:,\d{3}|[\s\xa0]000)` в `bot/utils/validators.py`.
