# План: извлечение одиночных слов из keyword-фраз

## Контекст и проблема

### Текущее поведение

При ручной смене категории траты бот сохраняет **всю очищенную фразу** как keyword:

- Исправили «Стамбул обед» на Путешествия → в БД `"стамбул обед"` → Путешествия.
- Исправили «Стамбул автобус» на Путешествия → в БД отдельная запись `"стамбул автобус"` → Путешествия.
- И так далее для каждой новой комбинации.

Поиск совпадений (`match_keyword_in_text` в [bot/utils/keyword_service.py:310](../bot/utils/keyword_service.py)) работает в два уровня:

- **Exact:** фраза целиком должна совпасть (±1 буква на всю строку).
- **Word:** поиск keyword как отдельного слова в тексте — **работает только если keyword состоит из одного слова** (строка 380).

В результате keyword-фраза `"стамбул обед"` (2 слова) бесполезна для текста `"стамбул продукты"` — ни один из двух уровней не срабатывает. AI не знает о предпочтениях пользователя и относит трату в «Продукты». Пользователь вынужден менять вручную каждую новую комбинацию.

### Подтверждение на данных пользователя (profile_id=1)

В БД у пользователя 13 keywords в категории «Путешествия» со словом «стамбул»: `"стамбул обед"`, `"стамбул проезд"`, `"стамбул чай"`, `"стамбул ужин"`, `"стамбул автобус"`, `"стамбул продукты"`, `"ноутбук стамбул"`, `"гостишка стамбул"`, `"стамбул подвал колоннами"`, и т.д. Все — фразы из 2+ слов. Общий токен `"стамбул"` нигде отдельно не сохранён.

### Корень проблемы

Логика обучения не выделяет **общий сигнал** из повторяющихся исправлений. Каждая новая комбинация воспринимается как новый уникальный keyword.

---

## Решение (вариант A с правками Codex)

Идея: при ручной смене категории трат **дополнительно к фразе** извлекать из неё отдельные слова-кандидаты и сохранять как самостоятельные keywords — но **только те, которые не конфликтуют** с keywords других категорий пользователя.

### Ключевые принципы

1. **Только при ручной коррекции.** AI-категоризация на создании траты не порождает одиночные keywords (иначе закрепляем ошибки AI).
2. **Мягкая уникальность.** Не удаляем существующие keywords других категорий — если слово уже занято, просто пропускаем кандидата.
3. **Проверка токена как подстроки-токена, а не только exact-row.** `"обед"` считается занятым, если встречается внутри любой keyword-фразы другой категории (например, `"обед сане"` → Кафе).
4. **Приоритет длинных фраз при матчинге.** Одиночное слово `"озон"` не затирает специфическое правило `"озон книги"` — при матчинге собираем ВСЕ совпадения и выбираем самое специфическое.
5. **Никакого negative feedback.** Одиночные слова сосуществуют с фразами; конфликты разрешаются приоритетом при матчинге, а не удалением.

### Как это решит сценарий пользователя

1. Пользователь исправляет «Стамбул обед» → Путешествия.
2. Сохраняется фраза `"стамбул обед"` → Путешествия (как сейчас).
3. **Новая логика:** разбиваем `["стамбул", "обед"]`:
   - `"обед"` — занят категорией Кафе (есть keyword `"обед сане"`) → **skip**.
   - `"стамбул"` — нигде не встречается → **создаём** одиночный keyword → Путешествия.
4. Следующая трата «Стамбул продукты»:
   - Матчер находит одиночный `"стамбул"` → Путешествия.
   - AI не вызывается, категория ставится автоматически.

### Как это защитит от ошибок

Сценарий с Озоном:
1. В БД: `"озон"` → Продукты (создано ранее из «Озон еда»).
2. Пользователь пишет «Озон книги». Матчер находит совпадение `"озон"` → Продукты. Пользователь исправляет на Развлечения.
3. Сохраняется фраза `"озон книги"` → Развлечения.
4. Попытка извлечь одиночные слова: `"озон"` уже занят Продуктами → **skip**. Одиночное `"книги"` — проверяем.
5. В БД: `"озон"` → Продукты, `"озон книги"` → Развлечения, возможно `"книги"` → Развлечения (если свободно).
6. Следующая трата «Озон молоко» → матчится только `"озон"` → Продукты (верно).
7. Следующая трата «Озон книги Рубакин» → матчится и `"озон"` (Продукты) и `"озон книги"` (Развлечения). Побеждает более специфичная фраза → Развлечения (верно).

## Детали реализации

### 1. Новые функции в `bot/utils/keyword_service.py`

**`extract_candidate_tokens(phrase: str) -> list[str]`**

Принимает уже очищенную фразу (результат `prepare_keyword_for_save`). Возвращает список слов-кандидатов для сохранения как одиночные keywords.

Правила фильтрации:
- Фраза должна содержать 2–4 слова (если 1 — это уже одиночный keyword, ничего делать не нужно; если >4 — фраза и так отбрасывается в `prepare_keyword_for_save`).
- Каждый кандидат:
  - длина ≥ 3 символов;
  - не состоит только из цифр;
  - не в `STOP_WORDS` (на всякий случай, хотя `prepare_keyword_for_save` уже очистил);
  - не содержит дефис внутри (такие слова как «бизнес-ланч» оставляем фразами целиком).

**`token_occurs_in_user_keywords(profile, token, exclude_category=None, is_income=False) -> bool`**

Проверяет, встречается ли токен в keywords пользователя (в любой категории, возможно кроме `exclude_category`).

Реализация:
- SQL-запрос через `keyword__iregex` с паттерном `(^|\s)<token>(\s|$)` — это поймает и exact-row `keyword = "обед"`, и вхождение `"обед сане"`, и `"стамбул обед"`. Регулярка в PostgreSQL ловит слово-токен независимо от позиции.
- Экранировать спецсимволы regex в токене (хотя после `prepare_keyword_for_save` там только буквы/цифры).
- Фильтр по `category__profile=profile` и опционально `category_id != exclude_category_id`.
- Возвращает `True` если найдена хоть одна запись.

**`add_single_word_keyword_soft(profile, category, word, is_income=False) -> tuple[object | None, str]`**

Мягкая версия `ensure_unique_keyword` специально для одиночных слов, извлечённых из фраз.

Логика:
1. Нормализуем слово через `prepare_keyword_for_save` (на выходе должно остаться ровно одно слово ≥3 символов).
2. Если токен занят в **других** категориях (`token_occurs_in_user_keywords(..., exclude_category=category)`) → вернуть `(None, "skipped_conflict")`.
3. Если точно такой же keyword уже существует в целевой категории → `usage_count += 1`, вернуть `(keyword, "incremented")`.
4. Если нигде нет → `get_or_create` в целевой категории, вернуть `(keyword, "created")`.

Логировать все три исхода через `logger.info` с префиксом `[SINGLE_WORD_KEYWORD]` — это нужно для мониторинга и последующего разбора.

### 2. Приоритизация при матчинге

**Текущая проблема.** В [bot/utils/expense_parser.py:920-950](../bot/utils/expense_parser.py#L920) код в цикле перебирает категории пользователя, для каждой — keywords, и на первом совпадении делает `break`. Порядок результатов из БД не детерминирован, поэтому одиночное слово `"озон"` может сработать раньше, чем длинная фраза `"озон книги"`.

**Решение — новая функция в `keyword_service.py`:**

**`find_best_keyword_match(profile, text, is_income=False) -> KeywordMatch | None`**

Где `KeywordMatch` — namedtuple или dataclass: `(keyword_obj, category, match_type, specificity_score)`.

Алгоритм:
1. Один запрос в БД на все keywords активных категорий пользователя (с `select_related('category')`).
2. Для каждого keyword вызываем `match_keyword_in_text(kw.keyword, text)`.
3. Собираем ВСЕ совпадения в список.
4. Сортируем по специфичности (убывание):
   - primary: количество слов в очищенном keyword (фраза из 2 слов > 1 слова);
   - secondary: длина keyword в символах;
   - tertiary: `usage_count` (при равных приоритетах побеждает проверенный).
5. Возвращаем топ-1 или `None`.

**Рефакторинг `expense_parser.py`.** В блоке категоризации расходов ([L868-960](../bot/utils/expense_parser.py#L868)) заменить вложенные циклы по пользовательским категориям на один вызов `find_best_keyword_match(profile, text_for_keywords)`. Если найдено — берём `match.category` и инкрементим `usage_count` / `last_used`. Аналогично для доходов ([L1515-1535](../bot/utils/expense_parser.py#L1515)).

**Важно:** сохранить текущее поведение матчинга по именам категорий (когда совпало само имя категории, а не keyword) — оно отдельное и не затрагивается. В приоритизации keyword-match'ей имена категорий участвуют только если они добавлены как keyword (сейчас нет).

### 3. Интеграция в `update_keywords_weights`

**Точка подключения:** [expense_bot/celery_tasks.py:1492-1564](../expense_bot/celery_tasks.py#L1492). После существующего вызова `ensure_unique_keyword` на полной фразе:

```
# Существующий код сохраняет фразу
kw_obj, created, removed = ensure_unique_keyword(
    profile=expense.profile,
    category=new_category,
    word=keyword,
    is_income=False,
)

# НОВОЕ: извлекаем одиночные слова
from bot.utils.keyword_service import (
    prepare_keyword_for_save,
    extract_candidate_tokens,
    add_single_word_keyword_soft,
)

cleaned_phrase = prepare_keyword_for_save(keyword)
tokens = extract_candidate_tokens(cleaned_phrase)

for token in tokens:
    obj, outcome = add_single_word_keyword_soft(
        profile=expense.profile,
        category=new_category,
        word=token,
        is_income=False,
    )
    logger.info(
        f"[SINGLE_WORD_KEYWORD] expense_id={expense_id} token='{token}' "
        f"category='{new_category.name_ru or new_category.name}' outcome={outcome}"
    )
```

Аналогично для income в `update_income_keywords_weights` (отдельная Celery-задача в том же файле около [L1906](../expense_bot/celery_tasks.py#L1906), только с `is_income=True` и `IncomeCategory`).

### 4. Про существующие keywords

Существующие 13 keywords пользователя вида `"стамбул X"` остаются в БД как есть. Никаких миграций / management commands не делаем.

Новая логика извлечения одиночных слов сработает при **следующем** ручном исправлении любой траты: сохранится очередная фраза через `ensure_unique_keyword` + автоматически создастся одиночный keyword `"стамбул"` → Путешествия. Начиная с этого момента все траты «Стамбул X» будут попадать в Путешествия автоматически.

То есть потребуется одно ручное исправление после деплоя — цена простоты реализации.

## Тесты

### Unit-тесты в `tests/test_keyword_service.py`

**Для `extract_candidate_tokens`:**
- `"стамбул обед"` → `["стамбул", "обед"]`.
- `"обед"` (1 слово) → `[]` (ничего извлекать не нужно).
- `"ab"` (2 буквы) → `[]` (слишком коротко).
- `"123"` (число) → `[]`.
- `"бизнес-ланч москва"` → `["москва"]` (дефис внутри слова отбрасывает).
- `"a b c d e"` (5 слов) → `[]` (фраза слишком длинная, в prepare_keyword_for_save она бы и не создалась).

**Для `token_occurs_in_user_keywords`:**
- Фикстура: профиль с `"обед сане"` → Кафе.
- `token="обед"`, `exclude_category=Путешествия` → `True`.
- `token="сане"`, `exclude_category=Путешествия` → `True`.
- `token="стамбул"`, `exclude_category=Путешествия` → `False`.
- `exclude_category=Кафе` → все три → `False` (исключили ту же категорию).
- Паттерн regex должен находить токен и в начале, и в середине, и в конце фразы.

**Для `add_single_word_keyword_soft`:**
- Пустая БД → `create` → `outcome="created"`.
- Такой же keyword уже есть в целевой категории → `outcome="incremented"`, `usage_count` увеличивается.
- Токен занят в другой категории (exact или как часть фразы) → `outcome="skipped_conflict"`, ничего не пишется в БД.
- Слово короче 3 символов → `None, "skipped_too_short"`.

**Для `find_best_keyword_match`:**
- В БД `"озон"` → Продукты и `"озон книги"` → Развлечения. Текст «Озон книги Рубакин» → побеждает `"озон книги"`.
- В БД только `"озон"`. Текст «Озон молоко» → матчится `"озон"`.
- Нет совпадений → `None`.
- При равной специфичности выигрывает больший `usage_count`.

### Интеграционные тесты

**Сценарий пользователя:**
1. Создать профиль с keyword `"обед сане"` → Кафе.
2. Создать expense «Стамбул обед» в Кафе (AI-предсказание).
3. Обновить категорию на Путешествия (симулировать `update_expense`).
4. Дождаться выполнения `update_keywords_weights` (в тесте — синхронно).
5. Проверить БД: должны существовать `"стамбул обед"` → Путешествия И `"стамбул"` → Путешествия. Keyword `"обед"` создаваться не должен.
6. Создать expense «Стамбул автобус» через `parse_expense_message` → проверить, что категория = Путешествия (через keyword `"стамбул"`), AI не вызывался.

## Риски и что проверить при ревью

1. **Производительность `find_best_keyword_match`.** Сейчас цикл разбит по категориям, у активных пользователей это 20–30 keywords. Новая функция загружает все сразу, сортирует в Python — остаётся в тех же порядках. БД-запрос один вместо N.
2. **`iregex` в PostgreSQL.** `iregex` использует POSIX regex — спецсимволы должны быть экранированы. Для наших токенов (буквы/цифры после нормализации) это не проблема, но добавить safeguard `re.escape(token)` не помешает.
3. **Случай с name-коллизией (не keyword, а имя категории).** `match_keyword_in_text` в [bot/utils/expense_parser.py:910](../bot/utils/expense_parser.py#L910) используется для проверки имени категории в тексте — эту часть НЕ трогаем, она остаётся отдельной веткой логики и срабатывает после провала `find_best_keyword_match`. Но: если пользователь назвал категорию «Стамбул», это имя будет матчиться наравне — нужно сверить приоритет (имя категории должно побеждать keyword-совпадение? или наоборот?). Текущее поведение — имя выигрывает (проверяется первым). Оставляем как есть.
4. **Что если фраза состоит только из `STOP_WORDS` и одного полезного слова.** Например, «купил в озоне» → после чистки `"озоне"`. `prepare_keyword_for_save` вернёт одно слово — это уже одиночный keyword, `extract_candidate_tokens` ничего не добавит. Всё ок.
5. **Что если пользователь переименует или удалит категорию, в которой был одиночный keyword.** Каскадное удаление/сохранение по FK — поведение как у существующих keywords, ничего нового.
6. **Locale/case.** `prepare_keyword_for_save` приводит к lowercase. `token_occurs_in_user_keywords` должен сравнивать тоже в lowercase (regex без `i` флага, но поле `keyword` в БД хранится в lowercase — это уже гарантия).

## Метрики успеха

- В логах `[SINGLE_WORD_KEYWORD]` видно, что при ручном исправлении создаются новые одиночные keywords (outcome=created) для редких слов (города, бренды).
- Токены, которые логика правильно НЕ создаёт: «обед», «чай», «продукты», «автобус» — у пользователя есть фразы с ними в разных категориях, ожидаем outcome=skipped_conflict.
- После первого ручного исправления трат со словом «Стамбул»: при последующих «Стамбул X» категория = Путешествия ставится автоматически (видно по отсутствию лога `[AI_CATEGORIZATION]` для таких трат).
- Нет увеличения количества ошибочных категоризаций у других пользователей (watched через количество ручных исправлений в первую неделю).
