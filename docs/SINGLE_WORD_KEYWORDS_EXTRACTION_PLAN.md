# План: извлечение одиночных слов из keyword-фраз

## Контекст и проблема

### Текущее поведение

При ручной смене категории траты бот сохраняет **всю очищенную фразу** как keyword:

- Исправили «Стамбул обед» на Путешествия → в БД `"стамбул обед"` → Путешествия.
- Исправили «Стамбул автобус» на Путешествия → в БД отдельная запись `"стамбул автобус"` → Путешествия.
- И так далее для каждой новой комбинации.

Поиск совпадений (`match_keyword_in_text` в [bot/utils/keyword_service.py:310](../bot/utils/keyword_service.py)) работает в два уровня:

- **Exact:** фраза целиком должна совпасть (±1 буква на всю строку).
- **Word:** поиск keyword как отдельного слова в тексте — **работает только если keyword состоит из одного слова** (строка 380).

В результате keyword-фраза `"стамбул обед"` (2 слова) бесполезна для текста `"стамбул продукты"` — ни один из двух уровней не срабатывает. AI не знает о предпочтениях пользователя и относит трату в «Продукты». Пользователь вынужден менять вручную каждую новую комбинацию.

### Подтверждение на данных пользователя (profile_id=1)

В БД у пользователя 13 keywords в категории «Путешествия» со словом «стамбул»: `"стамбул обед"`, `"стамбул проезд"`, `"стамбул чай"`, `"стамбул ужин"`, `"стамбул автобус"`, `"стамбул продукты"`, `"ноутбук стамбул"`, `"гостишка стамбул"`, `"стамбул подвал колоннами"`, и т.д. Все — фразы из 2+ слов. Общий токен `"стамбул"` нигде отдельно не сохранён.

### Корень проблемы

Логика обучения не выделяет **общий сигнал** из повторяющихся исправлений. Каждая новая комбинация воспринимается как новый уникальный keyword.

---

## Решение

Идея: при ручной смене категории трат **дополнительно к фразе** извлекать из неё отдельные слова-кандидаты и сохранять как самостоятельные keywords — но только те, что подтверждены **минимум 2 keyword-фразами** в целевой категории и не конфликтуют с keywords других категорий.

### Ключевые принципы

1. **Только при ручной коррекции.** AI-категоризация на создании траты не порождает одиночные keywords (иначе закрепляем ошибки AI).
2. **Минимум 2 подтверждения (min-support).** Одиночный keyword создаётся только если токен встречается ≥ 2 keyword-фраз в целевой категории (включая только что сохранённую). Защищает от ложных обобщений у новых пользователей с пустой БД.
3. **Мягкая уникальность.** Не удаляем существующие keywords других категорий — если токен уже занят где-то, просто пропускаем кандидата (`skipped_conflict`).
4. **Проверка занятости — по токен-границе внутри фраз.** `"обед"` считается занятым не только по exact-row `keyword = "обед"`, но и если встречается как отдельный токен внутри любой keyword-фразы другой категории (например, `"обед сане"` → Кафе).
5. **Приоритет длинных фраз при матчинге.** Одиночное слово `"озон"` не затирает специфическое правило `"озон книги"` — при матчинге собираем ВСЕ совпадения и выбираем самое специфическое (число слов → символы → usage_count → id).
6. **Никакого negative feedback.** Одиночные слова сосуществуют с фразами; конфликты разрешаются приоритетом при матчинге, а не удалением.
7. **Атомарность.** Check + create одиночного keyword — внутри `transaction.atomic()` с `select_for_update` на записях `CategoryKeyword` целевой категории, чтобы исключить race condition при конкурентных исправлениях.

### Как это решит сценарий пользователя (profile_id=1)

У пользователя уже 13 фраз со «стамбул» в Путешествиях.

1. Пользователь исправляет «Стамбул кофе» → Путешествия.
2. Сохраняется фраза `"стамбул кофе"` → Путешествия (как сейчас).
3. **Новая логика:** для каждого токена кандидата проверяем min-support в целевой категории:
   - `"стамбул"` — встречается в 14 фразах (13 старых + 1 новая) ≥ 2 → кандидат.
   - Проверяем занятость в других категориях — нигде больше нет → **создаём** одиночный `"стамбул"` → Путешествия.
   - `"кофе"` — встречается в 1 фразе < 2 → **skip** (insufficient_support), либо вообще конфликт с другой категорией → skip.
4. Следующая трата «Стамбул продукты»:
   - Матчер находит одиночный `"стамбул"` → Путешествия.
   - AI не вызывается, категория ставится автоматически.

Для нового пользователя (пустая БД) эффект отложенный на одно исправление: при первом «Стамбул обед» → Путешествия создаётся только фраза `"стамбул обед"`, одиночный `"стамбул"` ждёт подтверждения. При втором «Стамбул автобус» → Путешествия — токен `"стамбул"` теперь в 2 фразах, создаётся одиночный `"стамбул"` → Путешествия. Дальше автоматически.

### Как это защитит от ошибок

Сценарий с Озоном:
1. В БД: `"озон еда"` и `"озон молоко"` → Продукты, одиночный `"озон"` → Продукты (подтверждён).
2. Пользователь пишет «Озон книги». Матчер находит одиночный `"озон"` → Продукты. Пользователь исправляет на Развлечения.
3. Сохраняется фраза `"озон книги"` → Развлечения через `ensure_unique_keyword` (фраза новая, уникальная).
4. Попытка извлечь одиночные слова: `"озон"` уже занят Продуктами (как exact-row и как часть других фраз) → **skip_conflict**. `"книги"` — встречается только в этой одной фразе Развлечений < 2 → **skip_insufficient_support**.
5. В БД: `"озон"` → Продукты, `"озон книги"` → Развлечения. Одиночный `"книги"` не создан.
6. Следующая трата «Озон молоко» → матчится `"озон"` → Продукты (верно).
7. Следующая трата «Озон книги Рубакин» → матчится и `"озон"` (Продукты), и `"озон книги"` (Развлечения). Побеждает более специфичная фраза → Развлечения (верно).

## Детали реализации

### 1. Новые функции в `bot/utils/keyword_service.py`

**`extract_candidate_tokens(phrase: str) -> list[str]`**

Принимает уже очищенную фразу (результат `prepare_keyword_for_save`). Возвращает список слов-кандидатов для сохранения как одиночные keywords.

Правила фильтрации:
- Фраза должна содержать 2–4 слова (если 1 — это уже одиночный keyword, ничего делать не нужно; если >4 — фраза и так отбрасывается в `prepare_keyword_for_save`).
- Каждый кандидат:
  - длина ≥ 3 символов;
  - не состоит только из цифр;
  - не в `STOP_WORDS`;
  - не содержит дефис внутри (`"бизнес-ланч"` оставляем фразами целиком).
- Приведение регистра и нормализация ё/е **не делаются** в этой функции — keywords хранятся в lowercase через `prepare_keyword_for_save`, никакой нормализации ё/е в проекте нет, и для одиночных слов её тоже не вводим (согласованность со старыми keywords).

**`token_occurs_in_user_keywords(profile, token, exclude_category=None, is_income=False) -> bool`**

Проверяет, встречается ли токен как **отдельное слово** в keywords пользователя (в любой категории, возможно кроме `exclude_category`).

Реализация:
- Обязательно `re.escape(token)` перед подстановкой в regex.
- Паттерн: `(^|[^\wа-яА-ЯёЁ])<ESC_TOKEN>([^\wа-яА-ЯёЁ]|$)` — границы по не-словным символам, явно включая кириллицу. Использовать `\s` нельзя: после очистки в БД могут попасть соседние не-пробельные разделители (дефис на краю отфильтровали ещё в `normalize_keyword_text`, но подстраховаться).
- SQL: `CategoryKeyword.objects.filter(category__profile=profile, keyword__iregex=pattern)`; при `exclude_category` — `.exclude(category_id=exclude_category.id)`.
- Возвращает `True` если найдена хоть одна запись.
- Замечание: поле `keyword` в БД хранится lowercase (через `prepare_keyword_for_save`), токен на входе тоже lowercase — `iregex` тут страхует от артефактов, но по факту работает как case-sensitive.

**`count_token_in_category_phrases(profile, token, category, is_income=False) -> int`**

Считает, в скольки keyword-записях целевой категории токен встречается как отдельное слово.

- Тот же `re.escape(token)` и тот же token-boundary паттерн.
- SQL: `CategoryKeyword.objects.filter(category=category, keyword__iregex=pattern).count()`.
- Важно: считаем **записи** (и одиночные keywords с `keyword = token`, и фразы где токен — часть). Пример: если в Путешествиях уже есть одиночный `"стамбул"` + 13 фраз со «стамбул» → счётчик = 14.

**`add_single_word_keyword_soft(profile, category, word, min_support=2, is_income=False) -> tuple[object | None, str]`**

Мягкая версия `ensure_unique_keyword` специально для одиночных слов, извлечённых из фраз.

Логика (вся обёрнута в `transaction.atomic()`; в начале делается `select_for_update` на `CategoryKeyword.objects.filter(category__profile=profile)` — блокируем все keywords пользователя для check-then-create):

1. Нормализуем слово через `prepare_keyword_for_save` (на выходе должно остаться ровно одно слово ≥3 символов).
2. Если < 3 символов → `(None, "skipped_too_short")`.
3. `count = count_token_in_category_phrases(profile, word, category, is_income)`. Если `count < min_support` → `(None, "skipped_insufficient_support")`.
4. Если `token_occurs_in_user_keywords(profile, word, exclude_category=category, is_income)` → `(None, "skipped_conflict")`.
5. Если exact-row `keyword = word` уже есть в целевой категории → `usage_count += 1` (save update_fields) → `(keyword, "incremented")`.
6. Иначе → `get_or_create(category=category, keyword=word, defaults={'usage_count': 1, 'language': 'ru'})` → `(keyword, "created")`.

Логировать все исходы через `logger.info` с префиксом `[SINGLE_WORD_KEYWORD]` для мониторинга.

Примечание по `select_for_update`: для SQLite в тестах это no-op (Django это переживает), но на production PostgreSQL гарантирует серализацию конкурентных исправлений.

### 2. Приоритизация при матчинге

**Текущая проблема.** В [bot/utils/expense_parser.py:920-950](../bot/utils/expense_parser.py#L920) код в цикле перебирает категории пользователя, для каждой — keywords, и на первом совпадении делает `break`. Порядок результатов из БД не детерминирован, поэтому одиночное слово `"озон"` может сработать раньше длинной фразы `"озон книги"`.

**Решение — новая функция в `keyword_service.py`:**

**`find_best_keyword_match_sync(profile_id, text, is_income=False) -> KeywordMatchDTO | None`**

Где `KeywordMatchDTO` — dataclass с **plain-значениями** (не lazy-моделями):

```python
@dataclass(frozen=True)
class KeywordMatchDTO:
    keyword_id: int
    keyword_string: str
    category_id: int
    category_name: str         # уже готовое отображаемое имя (name_ru/name_en с эмодзи)
    match_type: str            # "exact" | "word"
    specificity_score: tuple   # (word_count, char_count, usage_count, keyword_id)
```

Это важно: dataclass отдаётся наружу в async-код, там к нему обращаются без ORM-вызовов. Вся работа с ORM остаётся внутри `find_best_keyword_match_sync`.

Алгоритм:
1. Один SQL-запрос: `CategoryKeyword.objects.filter(category__profile_id=profile_id, category__is_active=True).select_related('category')`. Для доходов — аналогичный через `IncomeCategoryKeyword`.
2. Материализовать результат через `list(...)` внутри той же функции (не возвращать queryset!).
3. Для каждого keyword вызываем `match_keyword_in_text(kw.keyword, text)`.
4. Собираем ВСЕ совпадения в список.
5. Сортируем по убыванию специфичности с детерминированными tie-breakers:
   - primary: количество слов в очищенном keyword (фраза из 2 слов > 1 слова);
   - secondary: длина keyword в символах;
   - tertiary: `usage_count` (проверенное правило побеждает);
   - quaternary: `keyword_id` (стабильный порядок при полном равенстве).
6. Возвращаем топ-1 DTO или `None`.

**Async-обёртка.** В `expense_parser.py` (async-код) вызывать через:
```python
from asgiref.sync import sync_to_async
_find_best = sync_to_async(find_best_keyword_match_sync)
match = await _find_best(profile.id, text_for_keywords, is_income=False)
```

Важно: передаём `profile.id` (int), не объект — внутри функции получаем `Profile` заново через `get()` или работаем напрямую через фильтр по `category__profile_id`. Это полностью исключает `SynchronousOnlyOperation`.

Инкремент `usage_count` / `last_used` при попадании — отдельная короткая `sync_to_async` функция, принимает `match.keyword_id`.

**Рефакторинг `expense_parser.py`.** В блоке категоризации расходов ([L868-960](../bot/utils/expense_parser.py#L868)) заменить вложенные циклы по пользовательским категориям на один вызов `find_best_keyword_match_sync` через sync_to_async. Аналогично для доходов ([L1515-1535](../bot/utils/expense_parser.py#L1515)).

**Важно:** сохранить текущее поведение матчинга по именам категорий (когда совпало имя категории, а не keyword) — оно отдельное и не затрагивается. В приоритизации keyword-match'ей имена категорий участвуют только если добавлены как keyword (сейчас нет).

### 3. Интеграция в `update_keywords_weights`

**Точка подключения:** [expense_bot/celery_tasks.py:1492-1564](../expense_bot/celery_tasks.py#L1492).

Вся логика после существующего вызова `ensure_unique_keyword` (который сохраняет фразу) должна выполняться внутри **того же транзакционного блока**, что и создание одиночных keywords:

```python
from django.db import transaction

with transaction.atomic():
    # Существующий код сохраняет фразу
    kw_obj, created, removed = ensure_unique_keyword(
        profile=expense.profile,
        category=new_category,
        word=keyword,
        is_income=False,
    )

    # НОВОЕ: извлекаем одиночные слова
    from bot.utils.keyword_service import (
        prepare_keyword_for_save,
        extract_candidate_tokens,
        add_single_word_keyword_soft,
    )

    cleaned_phrase = prepare_keyword_for_save(keyword)
    tokens = extract_candidate_tokens(cleaned_phrase)

    for token in tokens:
        obj, outcome = add_single_word_keyword_soft(
            profile=expense.profile,
            category=new_category,
            word=token,
            min_support=2,
            is_income=False,
        )
        logger.info(
            f"[SINGLE_WORD_KEYWORD] expense_id={expense_id} token='{token}' "
            f"category='{new_category.name_ru or new_category.name}' outcome={outcome}"
        )
```

`add_single_word_keyword_soft` внутри делает свой `select_for_update` — он вкладывается в существующий `transaction.atomic()` (Django использует savepoint'ы). Это ок.

Аналогично для income в `update_income_keywords_weights` (отдельная Celery-задача в том же файле около [L1906](../expense_bot/celery_tasks.py#L1906)) с `is_income=True` и `IncomeCategory`.

### 4. Про существующие keywords

Существующие 13 keywords пользователя вида `"стамбул X"` остаются в БД как есть. Никаких миграций / management commands не делаем.

Новая логика сработает при **следующем** ручном исправлении любой траты со словом «Стамбул»: сохранится фраза через `ensure_unique_keyword`, а токен `"стамбул"` сразу выполнит условие min-support (13 старых фраз + 1 новая = 14 ≥ 2) → создастся одиночный keyword `"стамбул"` → Путешествия. Начиная с этого момента все траты «Стамбул X» будут попадать в Путешествия автоматически.

То есть потребуется одно ручное исправление после деплоя — цена простоты реализации.

## Тесты

### Unit-тесты в `tests/test_keyword_service.py`

**Для `extract_candidate_tokens`:**
- `"стамбул обед"` → `["стамбул", "обед"]`.
- `"обед"` (1 слово) → `[]`.
- `"ab"` (2 буквы) → `[]` (слишком коротко).
- `"123"` (только цифры) → `[]`.
- `"бизнес-ланч москва"` → `["москва"]` (дефис внутри слова отбрасывает носителя).
- `"a b c d e"` (5 слов) → `[]` (фраза слишком длинная).
- **Mixed case:** `"Стамбул Обед"` — только после `prepare_keyword_for_save` (lowercase) идёт в эту функцию; отдельный тест что на непонижении регистра функция корректно не плодит дубликаты.

**Для `token_occurs_in_user_keywords`:**
- Фикстура: профиль с `"обед сане"` → Кафе, `"автобусы стамбул"` → Путешествия, `"кафе-бар"` → Досуг, `"молоко 3.2"` → Продукты.
- `token="обед"`, `exclude_category=Путешествия` → `True` (внутри `"обед сане"`).
- `token="сане"`, `exclude_category=Путешествия` → `True`.
- `token="стамбул"`, `exclude_category=Путешествия` → `False` (только в Путешествиях, но они исключены).
- `exclude_category=Кафе`: `token="обед"` → `False` (единственное вхождение в исключённой).
- **Substring collision:** `token="рис"`, БД содержит только `"рисунок"` → `False` (граница слова не совпала).
- **Regex metacharacters:** `token="c++"` (гипотетический) — после `re.escape` не ломает regex.
- **Hyphenated:** `token="бар"`, БД содержит `"кафе-бар"` → `True` (после нормализации в keyword_service дефис на границе был удалён, т.е. в БД `"кафе бар"`, токен матчится).
- **Cyrillic boundaries:** `token="час"`, БД `"часы"` → `False` (часть слова, не токен).
- **ё/е parity:** текущее поведение — не нормализуется, тест фиксирует что `"ёлка"` и `"елка"` НЕ считаются одинаковыми.

**Для `count_token_in_category_phrases`:**
- Фикстура: в Путешествиях `"стамбул обед"`, `"стамбул автобус"`, одиночный `"стамбул"`; в Кафе `"обед сане"`.
- `count(token="стамбул", category=Путешествия)` = 3.
- `count(token="обед", category=Путешествия)` = 1.
- `count(token="обед", category=Кафе)` = 1.
- `count(token="несуществующий", ...)` = 0.

**Для `add_single_word_keyword_soft`:**
- Пустая БД, `min_support=2` → `outcome="skipped_insufficient_support"`, ничего не пишется.
- В БД 1 фраза со «стамбул» в Путешествиях, вызов для «стамбул» → `skipped_insufficient_support`.
- В БД 2+ фразы со «стамбул» в Путешествиях, токен нигде больше не встречается → `outcome="created"`.
- Токен занят в другой категории (exact или часть фразы) → `outcome="skipped_conflict"`.
- Одиночный уже есть в целевой категории + 2 фразы с токеном → `outcome="incremented"`, `usage_count++`.
- Слово короче 3 символов → `outcome="skipped_too_short"`.

**Для `find_best_keyword_match_sync`:**
- В БД `"озон"` → Продукты (usage=5) и `"озон книги"` → Развлечения (usage=1). Текст «Озон книги Рубакин» → побеждает `"озон книги"` (больше слов).
- В БД только `"озон"`. Текст «Озон молоко» → матчится `"озон"`.
- Нет совпадений → `None`.
- **Equal-specificity ties:** в БД `"озон"` → Продукты (usage=3, id=10) и `"озон"` → Развлечения (создано искусственно, usage=3, id=20). Текст «Озон» → побеждает меньший id (детерминизм). На деле такой ситуации не бывает (строгая уникальность фраз), но проверяем tie-breaker.
- **Equal words/chars, разный usage:** `"кофе"` → Кафе (usage=10) и `"кофе"` → Продукты (usage=2, id>10). Побеждает usage.

## Риски и known limitations

1. **Производительность `find_best_keyword_match_sync`.** Сейчас цикл разбит по категориям, у активных пользователей это 20–50 keywords. Новая функция загружает все сразу одним SQL, сортирует в Python — остаётся в тех же порядках, БД-запрос один вместо N. Для пользователя с 500+ keywords может потребоваться limit/pagination, но таких случаев в проекте нет.
2. **`iregex` в PostgreSQL.** Все токены экранируются через `re.escape` перед подстановкой в паттерн. Границы определены явно как `[^\wа-яА-ЯёЁ]` — это безопасно для кириллицы и не зависит от локали БД. Тест `Regex metacharacters` страхует от регрессии.
3. **Имя категории как keyword.** `match_keyword_in_text` в [bot/utils/expense_parser.py:910](../bot/utils/expense_parser.py#L910) используется для проверки имени категории в тексте — эту часть НЕ трогаем, она остаётся отдельной веткой и срабатывает после провала `find_best_keyword_match_sync`. Приоритет: имя категории > keyword (текущее поведение, сохраняем).
4. **Фраза из одного смыслового слова.** Например, «купил в озоне» → после чистки `"озоне"`. `prepare_keyword_for_save` вернёт одно слово — это уже одиночный keyword через `ensure_unique_keyword`, `extract_candidate_tokens` возвращает пустой список. Ок.
5. **Удаление/переименование категории.** Каскадное поведение — как у существующих keywords, ничего нового.
6. **Known limitation: старые phrase keywords в неправильной категории могут затенять новые одиночные.** Если пользователь когда-то ошибочно отнёс несколько фраз в неверную категорию — одиночный keyword правильной категории всё равно создастся, но при матчинге более длинная фраза может победить. Это проблема качества исторических данных, а не механики. Документируем. При необходимости пользователь может вручную удалить неверные фразы через админку.
7. **ё/е не нормализуется.** Согласуется с текущим поведением проекта. `"ёлка"` и `"елка"` — разные keywords. Пользователь вводит один вариант → работает для него. Тест `ё/е parity` фиксирует это поведение.
8. **Конкурентность.** `transaction.atomic()` + `select_for_update` на уровне `CategoryKeyword.objects.filter(category__profile=profile)` обеспечивают корректность. На SQLite (локальные тесты) `select_for_update` — no-op, полагаемся на GIL для smoke-теста; на PostgreSQL production — полноценный row-level lock.

## Метрики успеха

- В логах `[SINGLE_WORD_KEYWORD]` видны разные outcomes — не только `created`, но и `skipped_insufficient_support`, `skipped_conflict` (значит защитные механизмы работают).
- Токены «обед», «чай», «продукты», «автобус» у нашего пользователя: outcome=`skipped_conflict` (уже есть в других категориях).
