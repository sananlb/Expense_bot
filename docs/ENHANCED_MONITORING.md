## 🏥 Автоматическая проверка здоровья системы

### Новая задача: `system_health_check`
**Запускается каждые 15 минут** и проверяет:

#### 🔍 Проверяемые компоненты:
- **База данных PostgreSQL** (время ответа, доступность)
- **Redis** (доступность, использование памяти)
- **Telegram API** (доступность, время ответа)
- **OpenAI API** (доступность, если настроен)
- **Google AI API** (доступность, если настроен)
- **Celery воркеры** (количество активных воркеров, размер очередей)

#### 💻 Системные метрики:
- **Свободное место на диске** (предупреждение < 1GB)
- **Использование памяти** (предупреждение > 90%)
- **Использование CPU** (предупреждение > 80%)

#### 🚨 Автоматические алерты:
Система автоматически отправляет уведомления администратору при:
- Статусе "unhealthy" или "degraded"
- Любых критических проблемах
- Превышении системных лимитов

### Пример алерта:
```
⚠️ Системный алерт

Статус: Degraded
Время: 14:23:15

Проблемы:
• Redis connection failed: timeout
• High memory usage: 92.1%
• OpenAI API returned status 429
```

## 🔄 Перекрёстный мониторинг серверов (добавлено 2026-02-19)

### Проблема
Все проверки здоровья работают на том же сервере. Если сервер полностью падает — никакие уведомления не приходят.

### Решение
Два проекта на разных серверах проверяют друг друга:

```
Expense Bot (176.124.218.53)  ←→  Nutrition Bot (45.93.201.222)
       │                                    │
       │  GET /health/ каждые 11 мин    │
       ├───────────────────────────────────→ │
       │                                    │
       │  GET /health/ каждые 11 мин        │
       │ ←──────────────────────────────────┤
```

### Параметры

| Параметр | Значение |
|----------|----------|
| Частота проверки | Каждые 11 минут |
| Таймаут HTTP запроса | 15 секунд |
| Алерт после | 2 подряд проваленных проверок |
| Повторы внутри одной проверки | 3 HTTP попытки с паузой 5 сек |
| Recovery уведомление | При первой успешной проверке после алерта |
| Антиспам | Один алерт на падение (флаг в Redis, TTL 24ч) |
| Очередь Celery | `monitoring`, приоритет 10 |

### Celery задача: `check_remote_server_health`

**Файл:** `expense_bot/celery_tasks.py`

- HTTP GET на `https://showmefood.duckdns.org/health/`
- Состояние хранится в Redis (ключи `remote_server:failures`, `remote_server:alert_sent`, `remote_server:down_since`)
- При падении: 🚨 алерт с указанием сервера, URL, времени и ошибки
- При восстановлении: ✅ уведомление с временем простоя

### Health endpoint

**URL:** `/health/` (файл `expense_bot/urls.py`)
**Ответ:** `{"status": "ok"}`
**Аутентификация:** Не требуется

## 🔐 Публичный HTTPS / webhook мониторинг (добавлено 2026-04-07)

### Что проверяется

Новая Celery задача `check_public_endpoint_health` каждые 15 минут:

- делает HTTPS запрос на публичный `PUBLIC_HEALTHCHECK_URL` с валидацией сертификата;
- проверяет срок действия текущего TLS сертификата;
- запрашивает `getWebhookInfo` у Telegram и смотрит:
  - совпадает ли зарегистрированный webhook URL с ожидаемым;
  - не появились ли `pending_update_count`;
  - не вернул ли Telegram свежий `last_error_message`.

### Когда шлётся алерт админу

- публичный endpoint недоступен 2 проверки подряд;
- сертификат истекает меньше чем через `PUBLIC_CERT_WARNING_DAYS` дней;
- сертификат истекает меньше чем через `PUBLIC_CERT_CRITICAL_DAYS` дней;
- Telegram недавно вернул webhook error;
- webhook URL в Telegram отличается от ожидаемого.

### Зачем это нужно

Этот мониторинг ловит именно тот класс инцидентов, при котором:

- Django/Celery внутри сервера ещё живы;
- Telegram уже не может доставлять обновления из-за SSL/webhook проблемы;
- бот снаружи фактически недоступен, хотя процессы локально выглядят нормальными.

| Параметр | Значение |

**Скрипты:**
- `scripts/backup_to_gdrive.sh` — основной скрипт
- `scripts/setup_backup_cron.sh` — установка cron

```
02:00 MSK - collect_daily_analytics      (сбор аналитики)

05:00 MSK - update_top5_keyboards        (обновление топ-5)
```

## 🔍 Troubleshooting

### Частые проблемы:

#### 1. Отчеты не приходят
```bash
# Проверить переменные окружения
echo $ADMIN_TELEGRAM_ID
echo $MONITORING_BOT_TOKEN

# Проверить логи
tail -n 100 logs/expense_bot.log | grep "admin_report"
```

#### 2. Проверка системы показывает ошибки
```bash
# Проверить доступность компонентов
python manage.py shell -c "
from expenses.views import HealthCheckView
view = HealthCheckView()
result = view.get(None)
print(result.content)
"
```

#### 3. Аналитика не собирается
```bash
# Проверить активность пользователей
python manage.py shell -c "
from expenses.models import Expense, UserAnalytics
from datetime import date, timedelta
yesterday = date.today() - timedelta(days=1)
expenses = Expense.objects.filter(expense_date=yesterday)
analytics = UserAnalytics.objects.filter(date=yesterday)
print(f'Expenses: {expenses.count()}, Analytics: {analytics.count()}')
"
```
