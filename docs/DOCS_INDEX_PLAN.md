# План: система индексации документации для expense_bot и nutrition_bot

**Дата:** 2026-04-12
**Версия:** 2 (после ревью Codex)
**Цель:** Дать Claude (и любому другому агенту) быстрый, структурированный и **надёжный** доступ ко всем `.md` файлам обоих проектов.

---

## 🎯 Проблема

Сейчас:
- В expense_bot 69 `.md` файлов в `docs/`
- В nutrition_bot ~90 `.md` файлов в `project_management/`
- Нет единой карты — Claude не знает что существует
- **Нет оси актуальности** — активные документы смешаны с закрытыми инцидентами и старыми миграциями
- **Нет однозначности при дубликатах** — `PII_REMOVAL_PLAN.md` vs `PII_REMOVAL_PLAN_ACTUAL.md`: Claude не знает какой читать
- Поиск идёт угадыванием ключевых слов через Grep

## 🎯 Решение

**Трёхчастная система:**
1. `INDEX.md` в каждом проекте — карта всех `.md` с **двухосевой** структурой (статус × тема)
2. Субагент `docs-indexer` — автоматически обновляет INDEX
3. Инструкция в `CLAUDE.md` — когда запускать субагента

---

## 📐 Ключевой принцип: двухосевая классификация

Главная ось — **статус актуальности**. Второстепенная — **тема**.

### Ось 1: статус (определяет порядок чтения)

| Статус | Значение | Читать Claude |
|--------|----------|---------------|
| `current` | Описывает как проект устроен **сейчас** | Всегда первым для контекста |
| `active` | Незавершённая работа (план, задача, POC) | Когда вопрос про эту работу |
| `reference` | Справочник, не меняется часто | По явному запросу |
| `historical` | Закрытое событие (инцидент, завершённая миграция, старый отчёт) | Только по явному запросу |
| `deprecated` | Устарело, есть замена | Никогда (кандидат на архив) |

**Важное правило:**
Миграции, инциденты, отчёты **по умолчанию → `historical`**, даже если недавние.
Исключение: миграция **в процессе** → `active`.

### Ось 2: тема (для группировки внутри статуса)

- architecture — сервера, инфраструктура, Docker, БД
- security — безопасность, PII, compliance, legal
- features — функционал бота, UX, категории, мультиязычность
- api — API, интеграции, webhook
- deployment — локальный запуск, SSH, деплой, команды сервера
- misc — всё остальное

### Правила описаний (однострочники)

- **Одна строка, 60-100 символов**
- **Конкретика, а не назначение:** "NL-сервер, Docker, PostgreSQL" вместо "инфраструктура проекта"
- **Ключевые слова для поиска важнее стиля**
- **Без воды:** запрещены слова "документация о...", "описание для...", "инструкция по..."
- **Даты в имени файла → в описании не повторять**

## 🧠 Как агент извлекает описание из файла

Приоритет источников (первый найденный → используем):

1. **Frontmatter `description:`** — если есть `---\ndescription: ...\n---`
2. **H1 + первый содержательный абзац** — пропустить метаданные (Дата, Статус, Автор), взять первый текст с смыслом
3. **H2-заголовки оглавления** — если в начале есть `## Содержание` с ссылками, извлечь ключевые разделы
4. **Первые 150 строк** — если всё выше не дало результата, прочитать 150 строк и выделить суть
5. **Отметка `[TODO: уточнить]`** — если после всех попыток описание неуверенное. Агент **обязан** вернуть список таких файлов в отчёте — человек поправит вручную.

**Запрет:** не использовать только первые 30 строк. Этого мало для больших технических документов которые начинаются с метаданных.

## 🧭 Как агент определяет статус файла

Порядок проверки:

1. **Имя файла:**
   - `INCIDENT_*`, `*_REPORT`, `POSTMORTEM_*` → `historical`
   - `MIGRATION_*` + дата в прошлом → `historical`
   - `*_PLAN.md`, `*_PLAN_ACTUAL.md` → `active` (по умолчанию)
   - `*_OLD.md`, есть файл с суффиксом `_ACTUAL` → `deprecated`

2. **Frontmatter `status:`** — если есть, перекрывает всё остальное

3. **Ключевые маркеры в первых 50 строках:**
   - "Статус: завершено / выполнено / closed" → `historical`
   - "Статус: в процессе / planning / in progress" → `active`
   - "Статус: актуально / current" → `current`
   - "Статус: устарело / deprecated" → `deprecated`

4. **Правила пар:**
   - Если существует `X_ACTUAL.md`, то `X.md` автоматически → `deprecated`
   - Если существует `X_PLAN.md` и `X_REPORT.md`, то `_PLAN` → `historical` (работа завершена, есть отчёт)

5. **По умолчанию** — `reference`

## 🤖 Субагент `docs-indexer`

### Расположение
`.claude/agents/docs-indexer.md` (глобально, на пользователя)

### Фронтматтер
```yaml
---
name: docs-indexer
description: Обновляет INDEX.md в expense_bot/nutrition_bot после создания, удаления, переименования или существенного смыслового изменения .md файлов. Вызывай при любом изменении в документационных папках.
model: sonnet
tools: Read, Glob, Edit, Write, Bash
---
```

### Параметры вызова

Агент принимает в `prompt`:

```
project: expense_bot | nutrition_bot
action: add | remove | rename | update | refresh
files: [список путей к затронутым файлам]  (опционально)
note: "краткий контекст для сложных случаев"   (опционально)
```

Если параметры не переданы → полный `refresh`.

### Пути

- `expense_bot` → `/Users/aleksejnalbantov/Desktop/expense_bot/docs/INDEX.md`
- `nutrition_bot` → `/Users/aleksejnalbantov/Desktop/Nutrition_bot/project_management/INDEX.md`

### Алгоритм работы

1. **Читает** текущий `INDEX.md` (если существует)
2. **Собирает** актуальный список `.md` через `Glob`:
   - `expense_bot/docs/**/*.md` исключая `INDEX.md`, `DOCS_INDEX_PLAN.md`
   - `nutrition_bot/project_management/**/*.md` исключая `INDEX.md`
   - Игнорирует папки `archive*/`, `.git/`, `venv/`, `node_modules/`
3. **Для каждого файла определяет:**
   - Статус (по алгоритму выше)
   - Тему (по ключевым словам в имени и содержимом)
   - Описание (по приоритету источников выше)
4. **Сравнивает с текущим INDEX.md:**
   - Новые файлы → добавить
   - Удалённые → убрать
   - Изменённые (действие `update`) → пересчитать описание и статус
   - **Не трогать** записи о файлах которые не менялись (стабильность)
5. **Сохраняет INDEX.md** через `Write` (полное перезаписывание безопасно — мы восстанавливаем всё из актуального состояния файловой системы)
6. **Возвращает отчёт**

### Формат отчёта

```
✅ INDEX.md обновлён (project: expense_bot)

Добавлено (2):
  + NEW_FEATURE_PLAN.md → active / features
  + INCIDENT_2026-04-15_DB_LOCK.md → historical / architecture

Удалено (1):
  - OLD_DEPRECATED_DOC.md

Статус изменён (1):
  ~ PROMOCODE_REWORK_PLAN.md: active → historical (найден REPORT, работа завершена)

⚠️ Требуется уточнение от человека (1):
  ? docs/FEATURES.md — описание неуверенное [TODO]

Итого: current=12, active=8, reference=15, historical=30, deprecated=4
```

### Что агент НЕ делает

- ❌ Не читает файлы из `archive*/`
- ❌ Не перестраивает категории без необходимости
- ❌ Не создаёт новые оси классификации сверх {current, active, reference, historical, deprecated}
- ❌ Не удаляет `.md` файлы с диска (только убирает из INDEX)
- ❌ Не переносит файлы в archive/ (эта работа — только вручную пользователем)

## 📜 Инструкция в CLAUDE.md

Добавить в `/Users/aleksejnalbantov/Desktop/expense_bot/CLAUDE.md` (и аналогично в nutrition_bot):

````markdown
## 📖 Индекс документации

**Карта всех .md файлов:** [docs/INDEX.md](docs/INDEX.md)

**КРИТИЧЕСКИ ВАЖНО:** Читай `INDEX.md` **первым**, прежде чем искать информацию через Grep.
В INDEX есть разделы по статусу:
- 🎯 **Current** — всегда читай первым для понимания текущего состояния
- 🚧 **Active** — незавершённая работа, читай если вопрос про неё
- 📚 **Reference** — справочники, только по явному запросу
- 🗄️ **Historical** — закрытые инциденты и миграции, **не читай без явной причины**
- ⚠️ **Deprecated** — устарело, **не читай вообще**

### Когда запускать субагента `docs-indexer`

**Обязательно:**
- ✅ После создания нового `.md` файла в `docs/` или `project_management/`
- ✅ После удаления `.md` файла
- ✅ После переименования `.md` файла
- ✅ После **существенного смыслового изменения** содержимого файла (статус поменялся, описание устарело, план стал отчётом)
- ✅ В конце сессии, если были любые изменения в `.md` файлах (action: refresh)

**Не нужно:**
- ❌ При косметическом редактировании (опечатки, форматирование, мелкие правки текста)

**Как определить "существенное изменение":**
Если одно из условий:
- Поменялся `Статус:` в шапке файла
- Добавлен/удалён крупный раздел (>20% объёма)
- Изменилось назначение файла (план → отчёт, актуальный → устаревший)

### Как вызывать

```
Task(
  subagent_type: "docs-indexer",
  description: "Update docs INDEX",
  prompt: "project: expense_bot\naction: add\nfiles: [docs/NEW_FILE.md]"
)
```

Или в конце сессии:
```
Task(
  subagent_type: "docs-indexer",
  description: "Refresh INDEX",
  prompt: "project: expense_bot\naction: refresh"
)
```

Агент работает автономно, не ест основной контекст, возвращает краткий отчёт.
````

## ⚠️ Риски и ограничения

### 1. Агент криво категоризирует файл
**Митигация:**
- Жёсткий список из 5 статусов, ничего сверх
- Чёткий алгоритм определения статуса (имя → frontmatter → маркеры → пары → default)
- `[TODO: уточнить]` для неуверенных случаев, человек правит вручную

### 2. INDEX устаревает при смысловых изменениях без переименования
**Митигация:**
- Инструкция в CLAUDE.md: запускать индексатор при **существенном** смысловом изменении
- `refresh` в конце сессии всегда — подстраховка

### 3. Несколько параллельных изменений .md → конфликт
**Митигация:**
- Агент всегда читает файлы из актуального состояния ФС, не из переданных параметров
- Полное перезаписывание INDEX (не инкрементальное) → детерминированно

### 4. Большие файлы → агент тратит много времени на чтение
**Митигация:**
- Читать только нужное количество строк (до 150), не весь файл
- При `refresh` — приоритизировать файлы изменённые с даты последнего INDEX (git mtime)

### 5. Дубликаты с похожими именами (пары `*_ACTUAL`, `*_PLAN` + `*_REPORT`)
**Митигация:**
- Явные правила пар в алгоритме определения статуса
- Запуск через `--apply` только после ручной проверки, если таких пар много

### 6. Обход archive/ и venv/
**Митигация:** Glob паттерны исключают эти папки явно

### 7. Зависимость от Obsidian / MCP
**Митигация:** Система не зависит от Obsidian вообще. INDEX.md — обычный `.md`, читается через Read/Grep. Obsidian может его открыть — бонус, не требование.
