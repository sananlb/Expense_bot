# Incident Report: бот не отвечает в Telegram

Дата инцидента: 2026-04-10  
Подготовлено: Codex  
Окружение: production server `176.124.218.53`  
Проект: `expense_bot`

## 1. Краткое описание

На production-сервере бот перестал отвечать на входящие сообщения в Telegram. По первоначальным симптомам webhook до сервера доходил нестабильно, а исходящие запросы из контейнера к Telegram Bot API периодически зависали по timeout.

В ходе работ были выявлены и исправлены несколько независимых инфраструктурных проблем:

1. Nginx проксировал на `localhost`, из-за чего часть запросов уходила в `::1` и падала с `Connection refused`.
2. Контейнер бота резолвил `api.telegram.org` в проблемный IP `149.154.166.110`, по которому HTTPS-соединение из контейнера зависало.
3. В коде запуска webhook сначала происходил `set_webhook`, а сервер уже слушал позже, что создавало окно для нестабильной регистрации и доставки webhook.

После этого был применён аварийный обходной путь: production был переведён из `webhook` в `polling`, чтобы исключить зависимость от внешней доставки webhook со стороны Telegram.

На момент последней проверки:

- webhook у Telegram был удалён;
- контейнер бота был запущен в режиме `polling`;
- в логах был подтверждён успешный старт polling;
- пользователь при этом всё ещё сообщал, что бот не отвечает, поэтому инцидент нельзя считать полностью закрытым.

## 2. Симптомы

Наблюдались следующие признаки:

- бот не отвечал пользователям в Telegram;
- в логах ранее фиксировался `TelegramNetworkError: HTTP Client says - Request timeout error`;
- `getWebhookInfo` у Telegram периодически показывал `last_error_message: Connection timed out`;
- `pending_update_count` у webhook периодически рос;
- публичный endpoint `https://expensebot.duckdns.org/webhook/` снаружи отвечал, но Telegram всё равно временами не доставлял обновления;
- из контейнера бот не всегда мог установить HTTPS-соединение к `api.telegram.org`.

## 3. Подтверждённые технические причины

### 3.1. Ошибка IPv6/localhost в Nginx

В live-конфигурации Nginx upstream был указан как:

```nginx
proxy_pass http://localhost:8001;
proxy_pass http://localhost:8000;
```

При этом приложения на сервере слушали только:

- `127.0.0.1:8000`
- `127.0.0.1:8001`

В логах Nginx была подтверждена ошибка вида:

```text
connect() failed (111: Connection refused) ... upstream: "http://[::1]:8001/webhook/"
```

Это означало, что `localhost` иногда резолвился в `::1`, а IPv6 listener на этих портах отсутствовал.

### 3.2. Нестабильный исходящий доступ контейнера к Telegram API

В контейнере `expense_bot_app` DNS для `api.telegram.org` возвращал IP `149.154.166.110`. Подключение к этому адресу по HTTPS из контейнера приводило к timeout.

После принудительного использования альтернативного IP `149.154.167.220` запросы к Telegram API из контейнера снова проходили успешно.

Фактически это выглядело как сетевая проблема на маршруте между сервером и конкретным IP Telegram.

### 3.3. Порядок старта webhook в приложении

В коде бота webhook регистрировался до гарантированного старта HTTP-сервера. Это создавало риск, что Telegram начнёт отправлять запросы на endpoint, который ещё не готов принимать трафик.

Этот участок был исправлен: теперь сначала стартует HTTP-сервер, и только потом вызывается `set_webhook(...)`.

## 6. Что было подтверждено проверками

### 6.1. Проверки по ingress

Подтверждено:

- `https://expensebot.duckdns.org/webhook/` публично отвечает;
- `curl` до webhook URL с хоста проходил;
- прямой POST на webhook URL возвращал `200`;
- после фикса Nginx ошибка с `http://[::1]:8001` перестала воспроизводиться.

### 6.2. Проверки по egress

Подтверждено:

- исходящие запросы к Telegram API из контейнера по одному из резолвящихся IP зависали;
- после принудительного использования `149.154.167.220` запросы к Bot API снова начинали работать;
- в логах контейнера появилась строка:

```text
Applying Telegram API host override: 149.154.167.220
```

### 6.3. Проверки по режиму polling

Были подтверждены логи старта:

```text
Бот запущен и готов к работе
Start polling
Run polling for bot @showmecoinbot id=8239680156 - 'Coins'
```

Также `getWebhookInfo` после удаления webhook показывал:

- `url: ""`
- `pending_update_count: 0`

Это означало, что Telegram больше не пытается доставлять обновления по старому webhook-маршруту.

## 7. Что не было окончательно подтверждено

Несмотря на исправления инфраструктуры и перевод в `polling`, пользователь после этого всё ещё сообщал, что бот не отвечает.

На момент остановки работ не было окончательно подтверждено:

1. получает ли polling новые входящие апдейты от Telegram в production после переключения;
2. не падает ли один из основных message handlers уже после получения update;
3. нет ли отдельной логики в middleware/router, из-за которой входящие сообщения молча игнорируются;
4. не связано ли отсутствие ответа уже с прикладной логикой, а не с сетью/webhook.

Иными словами: инфраструктурные проблемы были найдены и исправлены, но end-to-end работоспособность на пользовательском сообщении не была подтверждена как восстановленная.

## 8. Побочные действия в ходе диагностики

Во время диагностики были отправлены 2 тестовых сообщения в бот. После замечания пользователя оба сообщения были удалены.

После этого дополнительных видимых пользователю диагностических сообщений не отправлялось.

## 9. Текущий зафиксированный статус на момент остановки

На последнем зафиксированном шаге production-состояние было таким:

- Nginx исправлен и использует `127.0.0.1`;
- контейнер `expense_bot_app` пересоздан;
- в контейнере применяется временный DNS override для `api.telegram.org`;
- Telegram webhook удалён;
- бот запущен в `polling`;
- `getWebhookInfo` показывал пустой webhook и `pending_update_count=0`;
- пользователь по-прежнему сообщал, что бот не отвечает.

## 10. Следующие шаги

Для окончательного закрытия инцидента требуется:

1. Сразу после реального входящего сообщения пользователя снять live-логи контейнера бота.
2. Проверить, появляется ли в polling факт получения update.
3. Если update приходит, локализовать handler или middleware, на котором цепочка останавливается.
4. Если update не приходит даже в polling, повторно проверить Bot API с точки зрения `getUpdates` и сетевого доступа уже в runtime.
5. После восстановления ответа бота вернуть production в нормальный режим работы, если polling рассматривается только как аварийная мера.
