# Fix: Парсинг дохода берёт число из адреса вместо суммы

**Дата:** 2026-02-18
**Статус:** Реализовано
**Приоритет:** Высокий (реальный баг в продакшене)

---

## Проблема

Пользователь 348740371 ввёл:
```
+75000 аренда Кольская 8
```

**Ожидание:** доход 75 000 RUB, описание "аренда Кольская 8"
**Результат:** доход **8 RUB**, описание "75000 аренда Кольская"

Пользователь был вынужден вручную исправить сумму через `edit_income_193`.

---

## Анализ причины

### Поток обработки (до фикса)

```
ВХОД: "+75000 аренда Кольская 8"
        ↓
1. detect_income_intent() → TRUE (паттерн ^\+\d)
        ↓
2. parse_income_message()
   - Убирает "+" → "75000 аренда Кольская 8"
        ↓
3. extract_amount_from_patterns("75000 аренда Кольская 8")
   - Проверяет ~70 паттернов с валютами → НЕТ совпадений
   - Доходит до fallback-паттернов (строки 306-308):
     306: r'(\d+)\s*$'    ← "число в конце" → СОВПАДАЕТ С "8"!
     307: r'^(\d+)\s'     ← "число в начале" → поймало бы "75000", но не доходит
        ↓
4. Результат: amount=8, description="75000 аренда Кольская"
```

### Корень проблемы

Файл: `bot/utils/expense_parser.py`, строки 306-308

Fallback-паттерны без указания валюты идут в таком порядке:
```python
r'(\d+(?:[.,]\d+)?)\s*$',      # 306: число В КОНЦЕ строки
r'^(\d+(?:[.,]\d+)?)\s',        # 307: число В НАЧАЛЕ строки
r'\s(\d+(?:[.,]\d+)?)\s',       # 308: число В СЕРЕДИНЕ строки
```

Паттерн "конец строки" (306) стоит **раньше** паттерна "начало строки" (307).
Для текста `"75000 аренда Кольская 8"` первым срабатывает `\d+\s*$` и ловит `"8"`.

### Почему нельзя просто поменять порядок паттернов

Перестановка (начало перед концом) сломает парсинг **расходов**, где сумма обычно в конце:
- `кофе 300` → должно быть 300, а не ошибка
- `обед 1500` → должно быть 1500
- `такси 250` → должно быть 250

Для расходов приоритет конца строки правильный. Проблема только в **доходах**.

---

## Решение

### Подход: `_extract_leading_income_amount()` в `parse_income_message`

Добавлена функция-хелпер `_extract_leading_income_amount(text)`, которая извлекает
число из **начала** текста (после удаления `+`/`плюс`). Вызывается **до**
`extract_amount_from_patterns`, чтобы не допустить подхват хвостового числа.

### Что реализовано

1. **Ведущее число** — если текст начинается с числа, оно считается суммой
2. **Множители** — `тыс`, `к`, `млн`, `thousand`, `k`, `million` после числа
3. **Валюта** — `руб`, `$`, `€`, `₽`, `тенге` и все остальные — убирается из описания
4. **Разделители тысяч** — пробел (`75 000`), запятая (`75,000`), точка (`75.000.000`)
5. **Логирование `amount_source`** — в логах видно какая ветка извлекла сумму:
   - `leading_number` — ведущее число (новый путь)
   - `extract_patterns` — стандартный `extract_amount_from_patterns` (fallback)
   - `leading_number_with_date` / `extract_patterns_with_date` — если дата была удалена

### Затрагиваемые файлы

| Файл | Изменение |
|------|-----------|
| `bot/utils/expense_parser.py` | Добавлены `_INCOME_LEADING_CURRENCY_RE`, `_extract_leading_income_amount()`, интеграция в `parse_income_message` |

| # | Правило | Пример | Результат |

## Баги найденные и исправленные при тестировании

1. **Запятая как разделитель тысяч**: `75,000` парсился как `75.000` (Decimal). Добавлена проверка `^\d{1,3}(,\d{3})+$` для определения тысячного разделителя.
2. **`\b` после символов валют**: `$`, `€`, `₽` — не word characters, `\b` после них не срабатывает. Символы вынесены в отдельную ветку regex без `\b`.
3. **Точка как разделитель тысяч**: `10.000` и `10.000.000` не поддерживались. Добавлен паттерн `\d{1,3}(?:\.\d{3})+` как первая альтернатива в regex, определение через `re.match(r'^\d{1,3}(\.\d{3})+')` вместо подсчёта точек.
4. **Дата-парсер съедал точки-тысячи**: `+10.000 премия` → `extract_date_from_text` принимал "10.00" за дату. Исправлено перестановкой: ведущее число извлекается **до** парсера дат.
5. **UnboundLocalError `text_without_date`**: После перестановки ветка ведущего числа не определяла `text_without_date`, но переменная использовалась ниже. Исправлено инициализацией `text_without_date = text_for_parsing` перед ветвлением.
