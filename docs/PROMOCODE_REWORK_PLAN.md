# План доработки логики промокодов

## Цель

Сделать логику промокодов корректной, атомарной, согласованной с моделью данных и понятной для администрирования.

## 1. Уточнить бизнес-правило

- Зафиксировать, что означает промокод "на 5 устройств":
  - вариант A: на 5 разных пользователей;
  - вариант B: на 5 физических устройств;
  - вариант C: на 5 активаций внутри одной семьи или группы.
- Если нужен вариант A, текущая схема может использовать `max_uses = 5`.
- Если нужен вариант B, нужно проектировать отдельную сущность устройства, потому что сейчас система знает только пользователя (`Profile`), а не устройство.
- Зафиксировать ожидаемое поведение для промокодов с типом `days`:
  - можно ли активировать без существующей подписки;
  - нужно ли всегда продлевать активную подписку;
  - какой `Subscription.type` должен создаваться в таких случаях.

## 2. Исправить модель данных

- Добавить в `PromoCode.clean()` валидацию:
  - `valid_until >= valid_from`;
  - `max_uses >= 0`;
  - для `percent`: `0 <= discount_value <= 100`;
  - для `fixed`: `discount_value >= 0`;
  - для `days`: `discount_value > 0`.
- Вызывать `full_clean()` перед сохранением промокода.
- Проверить, нужен ли отдельный параметр для ограничения использований:
  - глобальный лимит;
  - лимит на пользователя;
  - лимит на устройство, если это требуется бизнесом.
- Привести `Subscription.payment_method` к фактическому использованию:
  - либо добавить `promo` в choices;
  - либо перестать использовать `promo` и выбрать единый согласованный способ хранения.

## 3. Сделать применение промокода атомарным

- Обернуть применение промокода в транзакцию.
- Внутри транзакции:
  - заблокировать строку промокода;
  - повторно проверить активность, срок действия и лимит использований;
  - повторно проверить, что пользователь ещё не использовал промокод;
  - создать `PromoCodeUsage`;
  - атомарно увеличить `used_count`.
- Убрать небезопасный паттерн:
  - `promocode.used_count += 1`
  - `await promocode.asave()`
- Использовать атомарное обновление через `F('used_count') + 1`.

## 4. Перевалидировать промокод на всех этапах flow

- Проверять промокод не только в момент ввода, но и:
  - при выборе подписки со скидкой;
  - перед созданием инвойса;
  - после успешной оплаты, перед фиксацией использования.
- Если промокод истёк, отключён или исчерпал лимит между шагами, корректно завершать flow и показывать пользователю понятное сообщение.
- Не полагаться на ранее сохранённый `active_promocode` в state без повторной проверки текущего состояния промокода.

## 5. Исправить логику бесплатных и `days`-промокодов

- Проверить корректность `payment_method` для бесплатных подписок и продлений по промокоду.
- Убедиться, что бесплатные активации и продления учитываются единообразно в коде, админке и аналитике.
- Привести к единой логике создание или продление подписки при:
  - `discount_type = days`;
  - 100% скидке по `percent` или `fixed`.
- Уточнить, допустимо ли продление активной подписки без создания новой записи истории, или всегда должна создаваться отдельная запись `Subscription`.

## 6. Усилить Django Admin

- Добавить `ModelForm` для `PromoCodeAdmin` с серверной валидацией.
- В форме явно пояснить смысл `max_uses`:
  - это глобальный лимит использований промокода, а не лимит устройств.
- Улучшить отображение статуса промокода:
  - учитывать `valid_from`;
  - учитывать `valid_until`;
  - учитывать достижение лимита использований.
- Исправить заполнение `created_by`, чтобы оно не опиралось на несуществующие или ненадёжные поля профиля.

## 7. Покрыть тестами критические сценарии

- Добавить тесты модели `PromoCode`:
  - валидные и невалидные даты;
  - граничные значения `discount_value`;
  - отрицательный `max_uses`;
  - некорректные значения для `days`.
- Добавить тесты promo-flow:
  - промокод истёк между вводом и покупкой;
  - промокод деактивирован между вводом и покупкой;
  - лимит использований закончился между вводом и покупкой;
  - повторное использование одним пользователем;
  - 100% скидка;
  - `days`-промокод с активной подпиской и без неё.
- Добавить тесты на конкурентность:
  - два пользователя одновременно пытаются использовать последний доступный use.
- Добавить тесты на админ-валидацию формы промокода.

## 8. Проверить совместимость с аналитикой и отчётами

- Проверить все места, где используется `payment_method`.
- Убедиться, что promo-подписки корректно попадают:
  - в аналитические задачи;
  - в отчёты;
  - в фильтры и списки админки.
- Убрать расхождения между моделью, роутерами, celery-задачами и админкой.

## 9. Критерии готовности

- Нельзя создать невалидный промокод через Django Admin.
- Нельзя превысить `max_uses` при параллельных активациях.
- Нельзя использовать промокод после истечения срока или деактивации, даже если пользователь начал flow раньше.
- `used_count` всегда совпадает с фактическим числом записей `PromoCodeUsage`.
- Логика `payment_method` консистентна в модели, коде и аналитике.
- Поведение "на 5 устройств" формально определено и реализовано в соответствии с зафиксированным бизнес-правилом.

## Порядок внедрения

1. Уточнить бизнес-смысл "5 устройств".
2. Исправить модель и admin-валидацию.
3. Исправить транзакции и атомарность promo-flow.
4. Добавить перевалидацию на всех этапах.
5. Привести в порядок `payment_method`.
6. Дописать тесты.
7. Проверить аналитику и регрессии.
