Статус: Done (реализовано, тесты зелёные)

## Симптом (production)

Пользователь записал трату с суммой в долларах. Бот ответил:

```
🧾 389 ₽ ($5.30)
💻 Мои проекты
💸 Потрачено сегодня: 930 ₽
```

В описании траты осталось слово **«доллара»** (исходный текст содержал что-то вроде
`Домен для впн 5.30 доллара`). Ожидаемое описание: `Домен для впн`.

## Корень проблемы

Файл: `bot/utils/expense_parser.py`, константа `AMOUNT_PATTERNS` (строки ~266-337).

Сумма извлекается функцией `extract_amount_from_patterns()`. Она удаляет из текста
**весь матч** (`text[:match.start()] + text[match.end():]`), включая слово-валюту.
Но русские словоформы валют в паттернах покрыты неполно:

- `доллар(?:ов)?` матчит только «доллар» и «долларов», но НЕ «доллара», «доллару»,
  «долларе», «долларом», «долларами», «долларах».
- Когда ни один валютный паттерн не сматчил, срабатывает последний fallback
  `\s(\d+(?:[.,]\d+)?)\s`, который удаляет только число → слово валюты остаётся в описании.

Проверка показала, что проблема системная и затрагивает почти все валюты с русскими
названиями (доллар, фунт, юань, драм, манат, сом, франк, реал, рубль в косвенных падежах
рубль→рублю/рублём, и т.д.). Уже корректно работают только те, где использован суффикс
`[а-я]*` (гривна: `гривн[а-я]*`) или перечислены все формы (йена: `йен[а-я]*`,
рубль: `рублей|рубля|рубль`).

Важно: **определение валюты** (`detect_currency` через `CURRENCY_PATTERNS`) работает
корректно (использует `(?<![а-яА-Я])доллар` без правой границы), поэтому сумма
конвертируется верно (389₽ = $5.30). Баг только в формировании описания.

## Предлагаемое решение (ФИНАЛ, после ревью Codex + требования пользователя)

**Требование пользователя:** совпадения по валютам должны быть ТОЧНЫМИ — никаких
приблизительных масок `[а-я]*`. Только явные перечисления реальных словоформ.

**Подход:** вынести точные русские словоформы каждой валюты в единый словарь-константу
`RU_CURRENCY_FORMS` и собирать из него альтернативу для регулярок. Это убирает
дублирование (DRY) и рассинхронизацию между `AMOUNT_PATTERNS` и `_LEADING_CURRENCY_RE`.

Каждой валюте — кортеж точных форм, упорядоченных ДЛИННЫЕ ПЕРВЫМИ (иначе regex обрывает
матч на коротком окончании: `гривна|гривнами` → сматчит «гривна», оставив «ми»).
К собранной альтернативе добавляется правая граница `(?![а-яёА-ЯЁ])`, чтобы не цеплять
омонимы-префиксы («сомнительный», «суммарно», «реально», «лес», «5-долларовый»).

Точные формы (проверены тестами):

- USD: доллар, доллара, доллару, долларом, долларе, доллары, долларов, долларам, долларами, долларах
- GBP: фунт, фунта, фунту, фунтом, фунте, фунты, фунтов, фунтам, фунтами, фунтах
- CNY: юань, юаня, юаню, юанем, юане, юани, юаней, юаням, юанями, юанях
- AMD: драм, драма, драму, драмом, драме, драмы, драмов, драмам, драмами, драмах
- AZN: манат, маната, манату, манатом, манате, манаты, манатов, манатам, манатами, манатах
- CHF: франк, франка, франку, франком, франке, франки, франков, франкам, франками, франках
- KGS: сом, сома, сому, сомом, соме, сомы, сомов, сомам, сомами, сомах
- UZS: сум, сума, суму, сумом, суме, сумы, сумов, сумам, сумами, сумах
- BRL: реал, реала, реалу, реалом, реале, реалы, реалов, реалам, реалами, реалах
- MDL: лей, лея, лею, леем, лее, леи, леев, леям, леями, леях
- RUB: рубль, рубля, рублю, рублём, рублем, рубле, рубли, рублей, рублям, рублями, рублях
- INR: рупия, рупии, рупию, рупией, рупиям, рупиями, рупиях, рупий
- TRY: лира, лиры, лире, лиру, лирой, лир, лирам, лирами, лирах
- JPY: йена/иена + формы
- AED: дирхам + формы
- UAH: гривна + формы, гривен
- Несклоняемые (оставить как есть, но с правой границей): евро, тенге, теньге, лари, сомони

Английские названия (`dollars?`, `pounds?`, `yuan`, …), ISO-коды (`usd`, `gbp`, …),
символы (`$`, `€`, …) и сокращения (`долл`, `грн`, `тнг`, `руб`) НЕ трогаем —
они уже точны и однозначны.

## Места правки

1. `AMOUNT_PATTERNS` (строки ~266-337) — три блока паттернов «число + валюта».
2. `_LEADING_CURRENCY_RE` (строки ~1220-1243) — валюта после ведущего числа
   («5 долларов кофе»). ТА ЖЕ проблема: `доллар(?:ов|а)?`, `фунт(?:ов)?` и т.д.

`CURRENCY_PATTERNS` (детекция валюты) НЕ трогаем — детекция работает корректно
(389₽ = $5.30 определилось верно).

## Замечания Codex и решения

- [HIGH] широкие основы `сом/сум/реал/ле` → **ПРИНЯТО**: точные списки форм + правая граница.
- [MEDIUM] `[а-я]*` не ловит `ё` (рублём) → **СНЯТО**: используем явные формы, включая `рублём`.
  Подтверждено: нормализации `ё→е` в парсере НЕТ.
- [MEDIUM] правка только AMOUNT_PATTERNS недостаточна → **ПРИНЯТО**: правим и `_LEADING_CURRENCY_RE`,
  тесты на оба пути. `CURRENCY_PATTERNS` не трогаем.
- [LOW] `доллар[а-я]*` ловит «5-долларовый» → **СНЯТО**: точные формы не ловят прилагательные.
- [LOW] не применять широкую маску единообразно → **ПРИНЯТО**: везде точные формы.

## Риски и edge cases (для ревью)

1. **`сом[а-я]*` (KGS)** — может перехватывать русское слово «сом» (рыба) или начало
   других слов. Сейчас `сом(?:ов)?` уже матчит «сом». Оценить, не станет ли хуже
   (например «5 сомнительных» → «5 сомн…»? — да, `сом[а-я]*` это сматчит). **Возможно
   стоит ограничить набор: `сом(?:а|у|ом|ы|ов|ах|ами)?` вместо `[а-я]*`.**
2. **`ле[а-я]*` (MDL, молдавский лей)** — слишком широкая основа, перехватит «лес»,
   «лекарство» и т.п. **НЕ использовать `ле[а-я]*`.** Лучше явный список форм:
   `ле[ейяю:]|леев|леями|леях` → аккуратно: `лей|лея|лею|лее|леев|леями|леях`.
3. **`доллар[а-я]*`** — теперь матчит прилагательное «5-долларовый». Это меняет поведение
   (раньше не матчилось). Для бота учёта трат кейс крайне редкий; определение валюты уже
   трактует «долларовый» как USD. Считаю риск приемлемым.
4. **Жадность `[а-я]*`** — не выходит за границы слова (кириллица), пробел/знаки её
   останавливают. Подтверждено тестами.
5. **Синхронизация** — есть параллельные структуры: `CURRENCY_PATTERNS`,
   `_NUMBER_ONLY_CURRENCY_RE`, регулярка на строках ~393 (`CURRENCY_WORDS_RE`?),
   ~1221 (leading-amount). В них словоформы покрыты ПО-РАЗНОМУ. Нужно решить: править
   только `AMOUNT_PATTERNS` (минимально достаточно для бага) или синхронизировать все.
   Рекомендация: в этой правке трогаем только `AMOUNT_PATTERNS` (источник бага удаления
   суммы), остальное — отдельной задачей, чтобы не раздувать diff и не ломать детекцию.

## Тестирование

Добавить в `tests/test_expense_parser.py` параметризованные кейсы на остаток словоформы:

```python
@pytest.mark.parametrize("text,expected_desc", [
    ("Домен для впн 5.30 доллара", "Домен для впн"),
    ("10 доллара кофе", "кофе"),
    ("5 долларом такси", "такси"),
    ("100 фунта подписка", "подписка"),
    ("50 юаня алиэкспресс", "алиэкспресс"),
    ("300 рублю", ""),  # рублю/рублём
])
def test_currency_wordform_not_left_in_description(text, expected_desc):
    result = parse_expense_message(text, ...)
    assert result["description"].strip() == expected_desc
```

Плюс регресс-проверка, что НЕ ломаются:
- `5 долларов`, `5$`, `5 usd`, `5 руб`, `500 рублей`, `5 долл`
- слитное `5000kzt`, `100лари`
- определение валюты `detect_currency` для тех же строк.
