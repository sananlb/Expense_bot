# План реализации: Лимиты на категории трат + общий лимит

**Статус:** Утверждён по UI, прошёл Codex-ревью; добавлен общий лимит (2026-06-13)
**Дата:** 2026-06-12, обновлён 2026-06-13

---

## 1. Суть фичи

Два вида месячных лимитов:

1. **Лимит на категорию** («Кафе: 10 000 ₽/мес») — процент в подтверждении траты,
   шкала в обзоре месяца, уведомление при превышении 100%.
2. **Общий лимит на все траты** («60 000 ₽/мес») — устанавливается в настройках,
   визуально выделенная шкала в обзоре месяца, уведомления при достижении **80% и 100%**.

Лимит **информирует, а не запрещает** — трата записывается всегда.

### Зафиксированные UI-решения (пользователь, 2026-06-12/13)

1. Точка входа для категорийных лимитов — только карточка категории. Сводный экран «Лимиты» — не в v1.
2. Прогресс в подтверждении траты — процент в скобках после названия категории: `🛒 Продукты (72%)`.
3. Отдельное сообщение при превышении 100% категорийного лимита.
4. Шкала в обзоре месяца («Бюджет») под каждой категорией с лимитом + процент цифрой справа; работает при пролистывании месяцев.
5. **Общий лимит:** кнопка в настройках выше «Семейного бюджета»; экран без цифр —
   только объяснение курсивом с 💡 (как работает, когда приходят уведомления, где
   отображается прогресс, что категорийные лимиты ставятся в меню категории);
   уведомления на 80% и 100%; в «Бюджете» — только визуально выделенная шкала,
   без дополнительной строки «Общий лимит: X / Y».

## 2. Модель данных

### Используем существующую модель `Budget` (`expenses/models.py:520`)

Модель есть с initial-миграции, таблица `expenses_budget` существует в проде, нигде не используется.

- `category` заполнена → категорийный лимит.
- `category IS NULL` → **общий лимит на все траты**.

### Семантика legacy-полей (фиксируем явно, по ревью Codex)

| Поле | Использование в v1 |
|---|---|
| `period_type` | Всегда `'monthly'`. Все выборки фильтруют `period_type='monthly'`. |
| `start_date` | Дата установки лимита. Для месяцев раньше `start_date` шкала не показывается. |
| `end_date` | Всегда `NULL` в v1. |
| `is_active` | `False` = лимит убран. |

### Миграция (схема + данные)

1. **Поле `currency`** (CharField(3), ISO-код) — валюта профиля на момент установки/изменения.
   В прогресс попадают только траты с `expense.currency == budget.currency`.
2. **Два частичных unique constraint:**
   ```python
   models.UniqueConstraint(
       fields=['profile', 'category'],
       condition=Q(is_active=True) & Q(category__isnull=False),
       name='unique_active_budget_per_category'
   ),
   models.UniqueConstraint(
       fields=['profile'],
       condition=Q(is_active=True) & Q(category__isnull=True),
       name='unique_active_total_budget_per_profile'
   )
   ```
3. **Индекс** по (profile, is_active).
4. **Data-миграция перед constraint** (ревью Codex): нормализовать существующие строки
   `expenses_budget` (ожидается 0), деактивировать дубликаты, заполнить `currency` из профиля.

### Решения по семантике (v1)

| Вопрос | Решение v1 |
|---|---|
| Период | Календарный месяц по `expense_date` (DateField — та же логика, что в существующих сводках; TZ уже учтена при создании траты, в профиле есть timezone). |
| Rollover | Нет. Сброс 1-го числа. |
| Валюта | Лимит в валюте профиля. При изменении суммы валюта обновляется до текущей валюты профиля. |
| Семейный бюджет | Оба вида лимитов личные, считают только траты владельца. В режиме семейного бюджета шкалы в обзоре месяца скрываются. |
| Подписка | Установка/изменение лимитов проверяется **в сервисном слое**. При истёкшей подписке лимиты видны read-only. |

### 3.5 Прогресс в подтверждении траты (только категорийный лимит)

Процент в скобках после названия категории в `format_expense_added_message`
(`bot/utils/expense_messages.py:75`):

```
🧾 350 ₽
🍔 Кафе и рестораны (72%)
```

- При превышении: `🍔 Кафе и рестораны (107% 🔴)`.
- Округление процентов — **floor** (не показываем «100%», пока `spent < limit`).
- Общий лимит в подтверждении траты **не показывается** (по решению пользователя —
  его прогресс живёт в меню «Бюджет» и в уведомлениях 80/100%).

### 3.6 Шкалы в обзоре месяца («Бюджет», `reports.py` → `show_expenses_summary`)

**Общий лимит** — визуально выделенная шкала **сразу под строкой расходов, без
дополнительной текстовой строки** (по решению пользователя 2026-06-13): более
«плотные» глифы (`█` / `░`, 12 сегментов против 10 у категорий) + процент + статусный эмодзи:

```
📊 Сводка за июнь 2026

💸 Расходы: 48 350 ₽
██████████░░ 80% 🟡

📊 По категориям:
  🍔 Кафе и рестораны: 7 200 ₽
  ▰▰▰▰▰▰▰▱▱▱ 72%
  🛒 Продукты: 12 350 ₽
  ▰▰▰▰▱▱▱▱▱▱ 41%
  🎮 Развлечения: 16 100 ₽
  ▰▰▰▰▰▰▰▰▰▰ 107% 🔴
  🚕 Такси: 2 700 ₽
```

Правила:
- Общая шкала: 12 сегментов `█`/`░`, статусный эмодзи: 🟢 <80%, 🟡 80–99%, 🔴 ≥100%
  (пороги совпадают с порогами уведомлений). Отдельной строки «Общий лимит: X / Y» нет —
  сумма лимита видна из шкалы и процента, расходы — строкой выше.
- Категорийная шкала: 10 сегментов `▰`/`▱`, процент справа, при >100% — полная шкала + 🔴.
- Заполнение = `min(N, floor(percent/N_step))`, процент — floor. Глифы одинаковой ширины.
- Категории без лимита — без шкалы.
- **Пролистывание месяцев:** шкалы показываются начиная с месяца `budget.start_date`;
  сумма лимита берётся текущая (историю изменений не храним — упрощение v1).
- **Без дополнительных запросов трат:** суммы уже посчитаны (`summary['total']` для общего,
  `by_category` для категорий). Нужен один запрос активных лимитов профиля.
- В режиме семейного бюджета (`household_mode=True`) все шкалы скрыты.
- В блоке «Остальные траты» шкал нет.

### 3.7 Уведомления о порогах

**Категорийный лимит — только пересечение 100%:**

```
🔴 Превышен лимит!

🍔 Кафе и рестораны
Лимит: 10 000 ₽
Потрачено: 11 100 ₽ (111%)
До конца месяца: 18 дней
```

**Общий лимит — пересечение 80% и пересечение 100%:**

```
🟡 Достигнуто 80% лимита трат

Потрачено: 48 350 / 60 000 ₽
Осталось: 11 650 ₽
До конца месяца: 18 дней
```

```
🔴 Превышен общий лимит трат!

Лимит: 60 000 ₽
Потрачено: 61 200 ₽ (102%)
До конца месяца: 18 дней
```

Механика (общая для обоих видов):
- Пересечение порога T: `spent_before < T*limit <= spent_after`. `spent_before` = `spent_after`
  минус сумма созданной траты, **только если** трата попадает в текущий месяц и совпадает
  по валюте с лимитом (траты задним числом за другие месяцы порог не триггерят).
- Одна трата может пересечь сразу оба порога общего лимита (80% и 100%) — отправляется
  только уведомление о старшем пороге (100%).
- Расчёт «до/после» — в том же `@sync_to_async`-блоке, что и агрегация.
- Точки отправки: ручной ввод (`expense.py`) и авто-траты recurring payments (celery-задача).
- **Принятый риск (v1):** при гонке одновременных трат или retry celery возможен дубль/пропуск
  уведомления. Persisted outbox — over-engineering: дубль безвреден, окно гонки — миллисекунды.
- Редактирование/удаление трат может «вернуть» расходы под порог — повторное уведомление
  при новом пересечении допустимо. Проценты всегда самокорректируются (свежая агрегация).

### 3.8 Что НЕ делаем в v1 (осознанно)

- Сводный экран «Лимиты» по категориям — добавим, если фича приживётся.
- Предупреждение на 80% для **категорийных** лимитов (только для общего).
- Лимиты в PDF-отчётах.
- Недельные/дневные периоды, rollover.
- Учёт трат членов семьи (household); история изменений суммы лимита.

## 4. Сервисный слой

Новый файл `bot/services/budget.py`. Общий и категорийный лимит — одна механика,
`category_id=None` означает общий лимит:

```python
@dataclass
class LimitStatus:
    budget: Budget
    spent: Decimal              # потрачено за месяц (включая текущую трату)
    percent: int                # floor-процент
    exceeded: bool              # spent >= limit
    crossed_thresholds: list[int]  # пороги, пересечённые этой тратой: [], [80], [100], [80, 100]

async def get_limit(profile_id: int, category_id: int | None) -> Budget | None
async def set_limit(profile_id: int, category_id: int | None, amount: Decimal) -> Budget
async def remove_limit(profile_id: int, category_id: int | None) -> bool
async def get_limit_status(profile_id: int, category_id: int | None,
                           expense=None) -> LimitStatus | None
async def get_active_limits_map(profile_id: int) -> dict[int | None, Budget]
    # category_id -> Budget (None = общий), для шкал в обзоре месяца (один запрос)
```

Ключевые требования (с учётом ревью Codex):

- **`set_limit`:** для категорийного — валидация владельца категории (паттерн `create_expense`);
  проверка подписки; затем **в одной транзакции**: `select_for_update()` по активным лимитам →
  деактивация всех найденных → создание новой строки. Не `update_or_create`.
  `IntegrityError` от constraint → один повтор.
- **`get_limit_status`:** принимает объект траты; вычитает её из `spent` для расчёта
  `crossed_thresholds` только при совпадении месяца и валюты. Пороги: `[100]` для
  категорийного, `[80, 100]` для общего.
- **Без Redis-кеша в v1** (сбой инвалидации гасил бы уведомления). Запрос лимитов —
  по индексу (profile, is_active), дёшево.
- Агрегация: `Sum('amount')` по profile + currency лимита + календарный месяц по `expense_date`;
  для категорийного — плюс фильтр category.
- Все ORM-вызовы — внутри одного `@sync_to_async` блока с `select_related` (async_patterns).
- При создании траты оба статуса (категорийный + общий) считаются за один вызов/блок,
  чтобы не плодить запросы.
