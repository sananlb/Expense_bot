# План реализации: Уведомления о подписке только в дневное время (10:00-21:00)

## Цель
Не беспокоить пользователей уведомлениями ночью (с 21:00 до 10:00 по их локальному времени).

## Архитектура решения

### 1. Утилита для проверки времени пользователя
**Файл:** `bot/utils/time_helpers.py` (новый файл)

**Функция:** `is_daytime_for_user(timezone_str: str) -> bool`
- Принимает timezone пользователя из Profile.timezone
- Возвращает True если сейчас 10:00-20:59 по времени пользователя
- Возвращает False если сейчас 21:00-09:59 (ночное время)
- Обрабатывает некорректные/отсутствующие timezone (fallback на UTC)

**Технические детали:**
- Использовать `pytz` для работы с таймзонами (IANA timezones)
- Получить текущее время в UTC (`timezone.now()`)
- Конвертировать в таймзону пользователя через `now.astimezone(tz)` (правильная обработка DST)
- Проверить час: `10 <= hour < 21` (явная семантика: 10:00:00-20:59:59 = день, 21:00:00+ = ночь)
- Кешировать невалидные timezone в памяти (`set()`) чтобы логировать warning только один раз, избегая спама в логах

### 2. Модификация функции отправки уведомлений
**Файл:** `bot/tasks/subscription_notifications.py`

**Функция:** `check_expiring_subscriptions(bot: Bot)` (строки 16-121)

**Изменения:**
1. Импорт: `from bot.utils.time_helpers import is_daytime_for_user` (прямой импорт)
2. Перед отправкой уведомления (строка ~59, перед формированием message)
3. Добавить проверку: `if not is_daytime_for_user(subscription.profile.timezone):`
4. Если False - логировать и пропускать пользователя (continue)
5. Если True - отправлять уведомление как обычно

**Логирование:**
- INFO: "User {telegram_id} skipped: nighttime in timezone {tz} (will notify in next check)"
- INFO: "Sent notification to user {telegram_id} at {local_time} ({tz})"

### 3. Тестирование

**Тестовые сценарии:**
1. Пользователь с timezone='Europe/Moscow' в 14:00 MSK → уведомление отправляется
2. Пользователь с timezone='Europe/Moscow' в 22:00 MSK → уведомление НЕ отправляется
3. Пользователь с timezone='America/New_York' в 09:00 EST → уведомление НЕ отправляется
4. Пользователь с timezone='UTC' или None → fallback на UTC, проверка времени
5. Пользователь с некорректным timezone → fallback на UTC + логирование warning

**Ручное тестирование:**
- Создать тестовую подписку, истекающую завтра
- Запустить celery задачу вручную
- Проверить логи

## Технические детали

### Константы
```python
NOTIFICATION_START_HOUR = 10  # Начало дневного времени (включительно)
NOTIFICATION_END_HOUR = 21    # Конец дневного времени (НЕ включительно)
# Проверка: 10 <= hour < 21
# Дневное время: 10:00:00 - 20:59:59
# Ночное время: 21:00:00 - 09:59:59
```

### Fallback стратегия
Если timezone пользователя:
- Пуст (None) → использовать 'UTC'
- Некорректный (pytz.UnknownTimeZoneError) → использовать 'UTC'
- Кешировать невалидные timezone в памяти (module-level `set()`)
- Логировать warning только при ПЕРВОМ обнаружении невалидного timezone (избегать спама)

### Важные детали реализации
- **Преобразование времени:** Использовать `now.astimezone(pytz.timezone(tz))` для корректной обработки DST (Daylight Saving Time)
- **Импорт:** Прямой импорт `from bot.utils.time_helpers import is_daytime_for_user` (не через `__init__.py`)
- **"Завтра" по UTC:** Текущее поведение - проверка "истекает завтра" идет по серверному UTC. Это edge case для пользователей в крайних timezone (UTC+12/-12), но мы его НЕ трогаем в этой задаче

### Частота проверки
Задача запускается каждые 4 часа (строка 132 в subscription_notifications.py):
- 00:00, 04:00, 08:00, 12:00, 16:00, 20:00 (условно, зависит от запуска)
- Это означает что пользователь получит уведомление при ближайшей проверке в дневное время

## Примеры работы

### Сценарий 1: Пользователь в Москве (UTC+3)
- Подписка истекает завтра
- Проверка в 22:00 UTC (01:00 MSK) → пропустить
- Проверка в 02:00 UTC (05:00 MSK) → пропустить
- Проверка в 06:00 UTC (09:00 MSK) → пропустить
- Проверка в 10:00 UTC (13:00 MSK) → отправить ✓

### Сценарий 2: Пользователь в Нью-Йорке (UTC-5)
- Подписка истекает завтра
- Проверка в 14:00 UTC (09:00 EST) → пропустить
- Проверка в 18:00 UTC (13:00 EST) → отправить ✓

## Риски и ограничения

### Риск: Пользователь может не получить уведомление
**Сценарий:** Подписка истекает сегодня в 23:59, проверка в 22:00 (ночь для пользователя)
**Решение:** Проверки идут каждые 4 часа, до истечения подписки будет минимум 1-2 дневные проверки

### Риск: Некорректный timezone в БД
**Решение:** Fallback на UTC + логирование warning

### Ограничение: Задержка до 4 часов
**Приемлемо:** Уведомление за 1 день - не критично получить его с задержкой в несколько часов
