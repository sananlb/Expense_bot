```
Интернет
    │
    ▼
┌─────────────────────────────────────────────────┐

│  Разрешено: 22/tcp, 80/tcp, 443/tcp, 10050/tcp  │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  fail2ban (4 jail)                               │
│  - sshd: блокировка SSH-брутфорса               │
│  - nginx-bad-request: блокировка сканеров       │
│  - nginx-http-auth: защита HTTP-авторизации     │
│  - nginx-limit-req: блокировка при превышении   │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Nginx (порты 80/443)                            │
│  - SSL/TLS (Let's Encrypt)                       │
│  - Rate limiting по зонам                        │
│  - Security headers                              │
│  - Блокировка dotfiles (/.env, /.git, etc.)     │
│  - Блокировка расширений (.sql, .bak, .log)     │
│  - Проксирование на localhost:8000/8001/8010    │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Docker-контейнеры (только 127.0.0.1)            │
│  - expense_bot_app  → 127.0.0.1:8001            │
│  - expense_bot_web  → 127.0.0.1:8000            │

    │
    ▼

│  - AntiSpamMiddleware (защита от бот-атак)       │
│  - Rate limiter на уровне приложения             │
│  - Webhook secret token validation               │
```

**Уровни защиты (defense-in-depth):**

| Уровень | Что защищает | Компонент |

| 2 | Автоблокировка | fail2ban — банит IP после повторных нарушений |

| 5 | Логика приложения | AntiSpam, rate limiter, валидация |

## 3. Фаервол (UFW)

### Статус

**Включён:** 5 февраля 2026
**До этого:** UFW был **INACTIVE**, INPUT policy ACCEPT — все порты были открыты наружу

### Текущие правила

```
Status: active
Logging: on (low)
Default: deny (incoming), allow (outgoing), deny (routed)

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW IN    Anywhere           # SSH
80/tcp                     ALLOW IN    Anywhere           # HTTP
443/tcp                    ALLOW IN    Anywhere           # HTTPS
10050                      ALLOW IN    92.53.116.12       # Zabbix server 1
10050                      ALLOW IN    92.53.116.111      # Zabbix server 2
10050                      ALLOW IN    92.53.116.119      # Zabbix server 3
53/tcp                     ALLOW IN    Anywhere           # DNS (ранее настроено)
53/udp                     ALLOW IN    Anywhere           # DNS (ранее настроено)
```

### Что заблокировано

Все порты, кроме перечисленных выше, **закрыты** для входящих подключений. В частности:

| Порт | Сервис | Статус |
|------|--------|--------|
| 8000 | Django (expense_bot_web) | Закрыт снаружи |
| 8001 | Telegram bot (expense_bot_app) | Закрыт снаружи |
| 8010 | Django (ndfl_web) | Закрыт снаружи |
| 5432 | PostgreSQL | Закрыт снаружи (только внутри Docker) |
| 6379 | Redis | Закрыт снаружи (только внутри Docker) |

### Важное замечание о Docker и UFW

Docker модифицирует iptables напрямую через цепочку DOCKER, **обходя UFW**. Поэтому одного UFW недостаточно для блокировки Docker-портов. Решение — привязка портов к `127.0.0.1` в docker-compose (см. раздел 5).

### Блокировка чувствительных файлов (добавлено 5 февраля 2026)

```nginx
# Allow ACME challenges (Let's Encrypt cert renewal)
location ^~ /.well-known/ {
    allow all;
}

# Block dotfiles and dot-directories (except .well-known)
location ~ /\.(?!well-known/) {
    deny all;
}

# Block common backup/config artifacts by extension
location ~* \.(env|sql|bak|backup|log|ini|cfg)$ {
    deny all;
}
```

**Что блокируется:**

| Запрос | Результат | До изменений |
|--------|-----------|-------------|
| `/.env` | **403 Forbidden** | 404 (случайно) |
| `/.git/config` | **403 Forbidden** | 404 (случайно) |
| `/.htaccess` | **403 Forbidden** | 404 (случайно) |
| `/backup.sql` | **403 Forbidden** | 404 (случайно) |
| `/config.bak` | **403 Forbidden** | 404 (случайно) |
| `/debug.log` | **403 Forbidden** | 404 (случайно) |
| `/.well-known/acme-challenge/*` | **Разрешено** | Разрешено |

**Почему это важно:** До изменений файлы не отдавались потому что Django не знал маршрута (404). Но если бы кто-то добавил директиву `root` в nginx, указывающую на папку проекта, `.env` с токенами стал бы доступен. Теперь nginx **явно** блокирует такие запросы на своём уровне.

**Логирование сохранено** — попытки доступа видны в access.log для мониторинга.

## 5. Docker — изоляция портов

### Проблема (до 5 февраля 2026)

Docker-контейнеры были привязаны к `0.0.0.0` — все порты были доступны из интернета напрямую:

```yaml
# БЫЛО (уязвимо):
ports:
  - "8001:8000"   # Доступен с любого IP
  - "8000:8000"   # Доступен с любого IP
```

Это позволяло сканерам отправлять запросы напрямую в Django и бот, минуя nginx (и все его защиты — rate limiting, headers, блокировки).

### Решение (применено 5 февраля 2026)

Все порты привязаны к `127.0.0.1`:

```yaml
# СТАЛО (безопасно):
ports:
  - "127.0.0.1:8001:8000"   # Только localhost
  - "127.0.0.1:8000:8000"   # Только localhost
```

### Изменённые файлы

| Файл | Было | Стало |
|------|------|-------|
| `/home/batman/expense_bot/docker-compose.yml` (bot) | `"8001:8000"` | `"127.0.0.1:8001:8000"` |
| `/home/batman/expense_bot/docker-compose.yml` (web) | `"8000:8000"` | `"127.0.0.1:8000:8000"` |
| `/home/batman/ndfl/docker-compose.yml` (web) | `"8010:8000"` | `"127.0.0.1:8010:8000"` |

### Бэкапы

- `/home/batman/expense_bot/docker-compose.yml.backup_20260205_141331`
- `/home/batman/ndfl/docker-compose.yml.backup_20260205_141331`

### Результат проверки (снаружи)

| Порт | HTTP-код | Статус |
|------|----------|--------|
| `:8000` | 000 (connection refused) | Закрыт |
| `:8001` | 000 (connection refused) | Закрыт |
| `:8010` | 000 (connection refused) | Закрыт |

### Почему bind 127.0.0.1 надёжнее UFW для Docker

Docker обходит UFW, модифицируя iptables напрямую через цепочку DOCKER. Привязка к `127.0.0.1` работает **на уровне сокета** — порт физически не слушает на внешнем интерфейсе. Это защита независимо от состояния фаервола.

## 6. fail2ban

### Статус

```
Активные jail: 4

- nginx-http-auth

- sshd
```

### Конфигурация

**Общие настройки (`/etc/fail2ban/jail.local`):**

```ini
[DEFAULT]
bantime = 600         # 10 минут блокировки

ignoreip = 127.0.0.1/8 ::1
```

### Jail: nginx-bad-request

| Параметр | Значение |
|----------|----------|
| Лимит | 3 попытки за 60 секунд |
| Бан | 1800 секунд (30 минут) |
| Текущие баны | 0 |
| Всего банов | 0 |

**Фильтр** (`/etc/fail2ban/filter.d/nginx-bad-request.conf`):

```ini
[Definition]
failregex = ^<HOST>.*"UNKNOWN.*HTTP.*" 400
            ^<HOST>.*BadStatusLine
            ^<HOST>.*BadHttpMessage
            ^<HOST>.*Invalid method encountered
ignoreregex =
```

**Примечание:** После привязки Docker-портов к 127.0.0.1 большинство BadStatusLine ошибок исчезнет, т.к. сканеры больше не смогут подключиться к портам 8000/8001 напрямую.

## 15. Полезные команды

### Проверка безопасности

```bash
# Статус фаервола
sudo ufw status verbose

# Статус fail2ban
sudo fail2ban-client status
sudo fail2ban-client status sshd
sudo fail2ban-client status nginx-bad-request

# Забаненные IP
sudo fail2ban-client status sshd | grep "Banned IP"

# Последние баны
sudo tail -50 /var/log/fail2ban.log | grep -E 'Ban|Unban'

# Проверить открытые порты
ss -tlnp | grep -v '127.0.0'

# Последние SSH-логины
last -20

# Неудачные SSH-попытки
sudo journalctl -u sshd | grep "Failed" | tail -20
```

### Nginx

```bash
# Проверить конфигурацию
sudo nginx -t

# Перезагрузить
sudo nginx -s reload

# Проверить rate limiting в логах
sudo grep "limiting" /var/log/nginx/error.log | tail -20

# Проверить блокировки dotfiles
sudo grep "403" /var/log/nginx/access.log | tail -20
```

### Docker

```bash
# Проверить привязку портов
docker ps --format 'table {{.Names}}\t{{.Ports}}'

# Логи бота
cd /home/batman/expense_bot && docker-compose logs --tail=200 bot

# Статус контейнеров
cd /home/batman/expense_bot && docker-compose ps
```

### Управление fail2ban

```bash
# Разбанить IP
sudo fail2ban-client set sshd unbanip 1.2.3.4

# Забанить IP вручную
sudo fail2ban-client set sshd banip 1.2.3.4

# Добавить IP в whitelist
# Редактировать /etc/fail2ban/jail.local → ignoreip
sudo systemctl restart fail2ban
```

### Управление UFW

```bash
# Добавить правило
sudo ufw allow from 1.2.3.4 to any port 22

# Удалить правило
sudo ufw delete allow from 1.2.3.4 to any port 22

# Посмотреть правила с номерами (для удаления)
sudo ufw status numbered

# Удалить правило по номеру
sudo ufw delete 3
```
