# Исследование: разбиение голосового батч-ввода трат

**Статус:** Research / Design (реализация не начата)
**Дата:** 2026-06-17
**Автор:** Claude + консультация Codex

## 1. Проблема

Пользователь диктует голосом несколько покупок одним сообщением. Бот должен
разбить распознанный текст на отдельные траты. Текущая эвристика разбивает по
**каждому числу** и не отличает **количество с единицей измерения** от **суммы**.

### Реальные примеры (из логов, транскрипция Яндекса)

| Транскрипт | Результат | Ожидание |
|------------|-----------|----------|
| `Сыр нарезанный 275 р мороженое 110 Ручки 83 р` | 3 траты ✅ | 3 траты |
| `Рыба сушеная 290 кальмар 250 жигулевская 1 литр 180 р Мороженое 110` | **5 трат ❌** | **4 траты** |

Во втором случае «жигулевская **1 литр** 180 р» (пиво, 1 литр, 180 ₽) разорвалось
на «жигулевская 1» (число 1 принято за сумму) и «литр 180 р». Корень — число `1`
в `1 литр` это **количество/объём**, а splitter трактует любое число как границу/цену.

## 2. Текущий конвейер (voice → траты)

1. **STT** — `bot/services/voice_recognition.py`: для русского `primary=Yandex`
   (`bot/services/yandex_speech.py`), fallback=OpenRouter/Whisper; для остальных
   языков наоборот. Яндекс возвращает одну строку (~2.9 с).
2. **Split** — `bot/utils/multiple_expense_parser.py`: (a)
   `_split_by_explicit_separators` по `,` `;` перенос строки (приоритет);
   (b) `_split_voice_without_separators` — эвристика по числам по шаблону
   «описание число [описание число]…», пропуская суффиксы множителей/валют.
   Лимит `MAX_MULTIPLE_EXPENSES_PER_MESSAGE=10`.
3. **Parse** — `bot/utils/expense_parser.py:parse_expense_message()` вызывается на
   каждую часть (теперь параллельно: `asyncio.gather` + `Semaphore(5)`); каждая
   делает локальный парсинг суммы/валюты/описания + вызов AI-категоризации
   (DeepSeek ~4–5 с), если не сматчилось ключевое слово.

> Примечание: Whisper для русского НЕ используется (Яндекс в приоритете —
> подтверждено кодом и логами). Параллелизация парсинга уже внедрена и работает;
> к данной проблеме разбиения отношения не имеет.

## 3. Варианты решения (анализ Codex)

| Вариант | Решает «1 литр» | Скорость | Стоимость | Риски |
|---------|-----------------|----------|-----------|-------|
| **A. Эвристика + единицы измерения** | ✅ напрямую | мгновенно | бесплатно | разрастание правил (склонения, дроби, «2 по 150»), ложные срабатывания на названиях с числами |
| **B. Эвристика + AI при сомнении** | ✅ | быстро (типичный случай локально) | дёшево (AI только на спорных) | **лучшая прод-архитектура по Codex** |
| **C. Один мультимодальный вызов (Gemini Flash: аудио→структура)** | ✅ | менее предсказуемо | дороже | raw-аудио третьей стороне; один провайдер на STT+split = single point of failure; JSON может быть семантически неверным даже при валидной схеме; потеря Яндекс-приоритета |
| **D. Чисто эвристика** | ✅ (с правилами A) | мгновенно | бесплатно | длинные фразы (скидки, кол-ва без единиц, даты, «3 банки по 90») всё равно падают |

### Детали по Option C (Gemini Flash, аудио→всё одним вызовом)
Технически возможно (Gemini умеет аудио + structured JSON output). Но для
**русского** заменять Яндекс не оправдано: Яндекс уже быстр/точен на коротких
ru-OggOpus клипах. Главные риски — операционные: raw-аудио уходит ещё одной
третьей стороне, латентность менее предсказуема, сбой провайдера ломает сразу и
STT и разбиение, JSON может быть семантически неверным. **Рекомендация:** не
делать мультимодальный ИИ основным путём для русского; держать как
экспериментальный fallback (когда Яндекс вернул пусто/мусор) или A/B-тест.
- Gemini audio: https://ai.google.dev/gemini-api/docs/audio
- Gemini structured output: https://ai.google.dev/gemini-api/docs/structured-output
- Yandex STT (короткие OggOpus): https://aistudio.yandex.ru/docs/en/speechkit/stt/request.html

## 4. Вердикт Codex (рекомендуемая архитектура)

Оставить **Яндекс STT основным** для русского + улучшить локальный splitter с
учётом единиц измерения (protected spans) + дешёвый **детектор неоднозначности**
+ **один** text-only AI-вызов на разбиение **только** при обнаружении
неоднозначности. **Не** заменять Яндекс на мультимодальный ИИ как основной путь.

### Фазовый план
1. **Unit-aware эвристика** + пересмотр приоритета разделителей для voice:
   для `input_source="voice"` принимать split по запятым только если в каждом
   сегменте есть правдоподобная сумма (Яндекс превращает «и/плюс/ещё» в запятые →
   агрессивное дробление), иначе использовать числовой splitter.
2. Confidence/ambiguity-скоринг + логирование.
3. **AI text-split fallback** при неоднозначности (строгая JSON-схема + локальная
   валидация сумм/валют, чтобы не принять галлюцинации).
4. (опц.) AI-split возвращает `category_key`/подсказки → меньше вызовов DeepSeek
   (только при высокой уверенности или подтверждении по ключевым словам).
5. (опц.) Мультимодальный ИИ только как fallback/A-B-тест против Яндекса на
   реальных логах.

## 5. Детектор неоднозначности (когда звать AI-split)

Дешёвые сигналы (считать ДО per-item парсинга, валидировать ПОСЛЕ):
- кандидат-позиция без суммы;
- позиция начинается с unit-токена;
- два соседних числа, где первое сопровождается единицей, а второе не уверенно
  опознано как цена;
- число разбиений больше числа «ценовых» чисел;
- > 6 позиций при слабых разделителях;
- позиция с описанием < 2 токенов (кроме известных продуктов);
- всего числовых токенов на 2+ больше, чем опознанных цен;
- split по разделителям создаёт части без сумм;
- длинный транскрипт (например 120+ символов или 8+ чисел).

## 6. Единицы измерения и числовые паттерны (RU)

**Защищённые спаны** (число — это количество, не граница):
`число + unit`, `число + unit + продукт`, десятичные.

**Числа/количества:** `1`, `1.5`, `1,5`, `0.5`, `0,5`, `2x`, `2 х`, `2 по 150`,
плюс словами: `один/одна`, `два/две`, `пол`, `полтора/полторы`.

**Единицы (продукты):**
- объём: `л`, `л.`, `литр`, `литра`, `литров`; `мл`, `миллилитр(а/ов)`;
- вес: `кг`, `килограмм(а/ов)`; `г`, `гр`, `гр.`, `грамм(а/ов)`;
- штучные/упаковки: `шт`, `шт.`, `штука/штуки/штук`; `уп`, `уп.`,
  `упаковка/упаковки/упаковок`; `пачка/пачки/пачек`; `бутылка/бутылки/бутылок`;
  `банка/банки/банок`; `пакет/пакета/пакетов`; `кусок/куска/кусков`.

## 7. Связь с AI-категоризацией

AI-split (фаза 3) может вернуть `category_key`/подсказки и сократить вызовы
DeepSeek, но только при той же таксономии категорий и пороге уверенности.
Рекомендация для AI-split: возвращать только `description`, `amount`,
`currency`, `confidence`. Количество/объём остаётся частью `description`
(`жигулевское 1 литр`), отдельное поле `quantity` не нужно. Позже опционально
можно добавить `category_key`, и пропускать DeepSeek лишь при высокой
уверенности/подтверждении по ключевым словам. Иначе корректность разбиения
окажется связана с корректностью категоризации.

## 8. Затрагиваемые файлы (при реализации)

- `bot/utils/multiple_expense_parser.py` — unit-aware split, приоритет разделителей для voice, ambiguity-скоринг.
- `bot/utils/expense_parser.py` — согласование выбора суммы при наличии «число+единица».
- `bot/services/voice_recognition.py` / `yandex_speech.py` — без изменений (Яндекс остаётся primary); точка для возможного мультимодального fallback (фаза 5).
- (фаза 3) новый сервис AI-split + строгая схема/валидация.

## 9. Решение продукт-оунера

### 9.1. Сравнение с Nutrition_bot

В `C:\Users\_batman_\Desktop\Nutrition_bot` batch-разбор сделан иначе, чем в
Expense Bot:

- **Expense Bot:** сначала локальный splitter режет исходную строку на отдельные
  текстовые куски, затем каждый кусок отдельно парсится как трата
  (`bot/utils/multiple_expense_parser.py` → `parse_expense_message()`).
- **Nutrition_bot:** отдельного локального splitter-а нет. Голос после STT
  передаётся в text handler, затем один AI-вызов `parse_batch_food_text()`
  извлекает из всей исходной фразы структурированный JSON со списком блюд.
  После этого `calculate_nutrition_batch()` параллельно считает КБЖУ по каждому
  item.

Ключевые точки Nutrition_bot:

- `meals/handlers/voice_handler.py` — для RU сначала Yandex SpeechKit, затем
  fallback на OpenRouter; после транскрипции текст передаётся в обычный
  `process_text()`.
- `meals/handlers/text_handler.py` — `handle_food_record()` вызывает
  `nutrition_service.get_nutrition_from_text()`.
- `meals/unified_ai_service.py` — `parse_batch_food_text()` делает один
  text-only AI-вызов с `response_format={"type": "json_object"}` и возвращает
  `items`.
- `meals/ai_prompts.py` — промпт явно учит модель разделять блюда, понимать
  веса/объёмы и конвертировать `1 литр` / `1 л` / `500 мл` в граммы.
- `meals/models.py` — все распознанные блюда сохраняются в один `MealsBatch`,
  отдельные `Meal` привязаны к нему через `batch`.

Фактическое отличие: Nutrition_bot совмещает split и первичный структурный
парсинг в одном AI-шаге, а Expense Bot сейчас делает split локально до
семантического парсинга. Поэтому Nutrition_bot естественно переживает кейсы
вида `1 литр`, а Expense Bot ломается, если локальная эвристика принимает `1`
за сумму.

### 9.2. Проверка текущего Expense Bot

Проверка текущей реализации подтвердила баг из логов.

Вход:

```text
Рыба сушеная 290 кальмар 250 жигулевская 1 литр 180 р Мороженое 110
```

Текущий результат:

```python
[
    "Рыба сушеная 290",
    "кальмар 250",
    "жигулевская 1",
    "литр 180 р",
    "Мороженое 110",
]
```

Ожидаемый результат:

```python
[
    "Рыба сушеная 290",
    "кальмар 250",
    "жигулевская 1 литр 180 р",
    "Мороженое 110",
]
```

Причина подтверждена в `bot/utils/multiple_expense_parser.py`: текущий
`_split_voice_without_separators()` считает каждое число потенциальной суммой и
не защищает паттерн `число + единица измерения`.

### 9.3. Финальное решение по voice batch

Решение продукт-оунера: для платного голосового ввода сделать **AI audio→JSON
основным путём**, а связку **Yandex STT + DeepSeek text split** оставить
единственным fallback. Старый локальный voice splitter не использовать как
финальный fallback для голосового batch-ввода.

Основной premium voice pipeline:

```text
Telegram voice
→ audio-capable reasoning model (например Gemini Flash)
→ strict JSON: transcript + items
→ локальная валидация
→ подстановка валюты по профилю, если currency пустая
→ поиск старой цены для items без amount
→ создание трат
```

Fallback pipeline:

```text
Telegram voice
→ Yandex SpeechKit STT
→ DeepSeek text split в strict JSON
→ локальная валидация
→ подстановка валюты / старой цены
→ создание трат
```

Третьего fallback вида `Yandex STT → локальный splitter` не делаем. Если оба
AI-пути не дали валидный результат, бот просит пользователя повторить или
ввести траты текстом.

### 9.4. Минимальная JSON-схема

Нужны только 4 поля на item:

```json
{
  "description": "жигулевское 1 литр",
  "amount": 180,
  "currency": "RUB",
  "confidence": 0.94
}
```

Правила:

- `description` содержит всё, что относится к товару, включая количество,
  объём, упаковку: `1 литр жигулевское`, `сыр 300 г`, `2 пачки ручек`.
- `amount` — только цена, явно услышанная от пользователя. Если цена не
  прозвучала, вернуть `null`. Модель не должна придумывать, оценивать или
  восстанавливать цену.
- `currency` — только явно услышанная валюта. Если валюты нет в голосе,
  вернуть `null`; валюта подставляется локально из профиля пользователя.
- `confidence` — уверенность модели в разбиении именно этого item.

Желательно также возвращать общий `transcript`, чтобы в логах и валидации было
видно, что именно модель услышала:

```json
{
  "transcript": "рыба сушеная 290 кальмар жигулевское 1 литр 180 мороженое",
  "items": [
    {"description": "рыба сушеная", "amount": 290, "currency": null, "confidence": 0.95},
    {"description": "кальмар", "amount": null, "currency": null, "confidence": 0.82},
    {"description": "жигулевское 1 литр", "amount": 180, "currency": null, "confidence": 0.94},
    {"description": "мороженое", "amount": null, "currency": null, "confidence": 0.8}
  ]
}
```

### 9.5. Free vs premium

- **Free:** голосовой ввод недоступен. Пользователь вводит текстом; для batch
  ожидается нормальный явный разделитель, прежде всего запятая.
- **Premium:** голосовой ввод доступен и обрабатывается через основной
  audio→JSON pipeline с fallback на Yandex+DeepSeek.

### 9.6. Что остаётся обязательным

- Строгая локальная валидация JSON: типы полей, лимит items, положительные
  суммы, валидные валюты или `null`.
- Проверка, что `amount` не был придуман: цена должна быть явно представлена в
  transcript/audio-разборе. При сомнении item сохраняется с `amount=null`, после
  чего применяется поиск старой цены или уточнение у пользователя.
- Логирование `transcript`, raw JSON, выбранного pipeline и причины fallback.
- AI-категоризацию не смешивать с split-ом на первом этапе.

Итоговое решение: **для premium voice основным сделать Gemini/Voxtral-like
audio→expense JSON, fallback — Yandex STT + DeepSeek JSON split; для free оставить
только текстовый ввод с явными разделителями.**
